Variables & Data Types

The containers that hold all your data

What is a Variable?

A variable is a named container for storing data. Think of it like a labeled box — you put something inside, give it a name, and then use that name to get the value back later. In JavaScript there are three ways to create variables: var, let, and const.

let vs const vs var

const = the value will never change. Use this by default.
let = the value will change later (like a counter or a user's score).
var = old way, has confusing rules — avoid it completely.

// const — value never changes
const siteName = "My Portfolio";
const pi = 3.14159;

// let — value changes later
let score = 0;
score = score + 10; // now it's 10

// NEVER use var
var oldWay = "avoid this"; // confusing scoping rules

The 7 Data Types

JavaScript has 7 primitive types. You don't need to declare the type — JS figures it out automatically (this is called dynamic typing).

TypeExampleUse for
String"Hello" or 'Hello'Text, names, messages
Number42, 3.14, -10Prices, scores, ages
Booleantrue / falseOn/off switches, flags
NullnullIntentionally empty
UndefinedundefinedDeclared but not assigned
Object{ name: "Ali" }Collections of data
Array[1, 2, 3]Lists of items
const name    = "Myat";          // String
const age     = 19;              // Number
const isLoggedIn = true;         // Boolean
const empty   = null;            // Null
let  result;                     // Undefined (not assigned yet)
const user    = { id: 1 };       // Object
const colors  = ["red", "blue"]; // Array

// Check the type of anything with typeof
console.log(typeof name);     // "string"
console.log(typeof age);      // "number"
console.log(typeof isLoggedIn); // "boolean"

Real World Example

Think of an e-commerce cart. The product name is a string. The price is a number. isInStock is a boolean. The whole product is an object. The list of all products is an array.

❌ Common Mistakes

  • Using var — it has function scope instead of block scope, causing hard-to-find bugs
  • Trying to reassign a const — const name = "Ali"; name = "Bob"; throws an error
  • Confusing null and undefined — null is intentional emptiness, undefined means "not set yet"
  • Forgetting that typeof null returns "object" — this is a famous JS bug, not real behavior
  • Using the same variable name twice with const — it throws a "already been declared" error

Quick Summary

  • Use const by default
  • Use let when value changes
  • Never use var
  • 7 types: string, number, boolean, null, undefined, object, array
  • JS figures out the type automatically
  • Use typeof to check a type

Operators & Type Coercion

Math, comparisons, and JavaScript's weird automatic conversions

Arithmetic Operators

Used for math. The tricky one is % (modulo) — it gives you the remainder after dividing. For example 10 % 3 = 1 (because 10 ÷ 3 = 3 remainder 1). Useful for checking if a number is odd/even.

const a = 10;
const b = 3;

console.log(a + b);  // 13  (addition)
console.log(a - b);  // 7   (subtraction)
console.log(a * b);  // 30  (multiplication)
console.log(a / b);  // 3.33 (division)
console.log(a % b);  // 1   (remainder / modulo)
console.log(a ** b); // 1000 (power: 10³)

// Shorthand
let score = 0;
score += 10; // same as score = score + 10
score++;     // same as score = score + 1

Comparison Operators — === vs ==

This is critical. Always use === (triple equals). It checks both the value AND the type. == (double equals) only checks the value and does automatic type conversion, which causes surprising results.

// === checks value AND type (use this always)
console.log(5 === 5);     // true
console.log(5 === "5");   // false — different types!
console.log(5 === 6);     // false

// == only checks value (AVOID)
console.log(5 == "5");    // true — DANGEROUS!
console.log(0 == false);  // true — DANGEROUS!
console.log(null == undefined); // true — DANGEROUS!

// Other comparisons
console.log(10 > 5);   // true
console.log(10 >= 10); // true
console.log(5 !== 6);  // true (not equal)

Logical Operators

&& means AND — both must be true. || means OR — at least one must be true. ! means NOT — flips true to false.

const age = 20;
const hasID = true;

// AND — both conditions must be true
if (age >= 18 && hasID) {
  console.log("Allowed entry");
}

// OR — at least one must be true
const isAdmin = false;
const isMember = true;
if (isAdmin || isMember) {
  console.log("Can access dashboard");
}

// NOT — flips the boolean
const isLoggedIn = false;
if (!isLoggedIn) {
  console.log("Please log in");
}

Type Coercion — JavaScript's Automatic Conversions

JavaScript sometimes silently converts types for you. This is called type coercion and it causes bugs if you don't know about it.

// String + Number = String (concatenation wins)
console.log("5" + 3);    // "53"  ← NOT 8!
console.log("5" - 3);    // 2     ← subtraction converts to number

// Force type conversion manually (safer)
const input = "42";
const num = Number(input);  // 42
const str = String(42);     // "42"
const bool = Boolean(0);    // false (0, "", null, undefined = false)

❌ Common Mistakes

  • Using == instead of === — causes silent bugs with type coercion
  • Adding strings and numbers accidentally — "total: " + 5 + 3 gives "total: 53"
  • Not knowing "falsy" values: 0, "", null, undefined, NaN, false all evaluate to false
  • Using NaN === NaN — this returns false! Use isNaN(value) instead

Quick Summary

  • Always use === never ==
  • % gives the remainder
  • && = AND, || = OR, ! = NOT
  • String + Number = String (watch out)
  • Use Number() / String() to convert manually
  • Falsy values: 0, "", null, undefined, NaN, false

Functions

Reusable blocks of code — the backbone of every program

What is a Function?

A function is a reusable block of code that does a specific job. Instead of writing the same 10 lines over and over, you write it once inside a function, give it a name, and then call it whenever you need it. Real world: a microwave is a function — you put food in (input), press start (call it), get hot food out (return value).

3 Ways to Write Functions

There are three main styles. The most modern and popular is the arrow function. They all do the same thing — the difference is syntax and how they handle this.

// 1. Function Declaration (can be called before it's defined)
function greet(name) {
  return "Hello, " + name + "!";
}

// 2. Function Expression (stored in a variable)
const greet = function(name) {
  return "Hello, " + name + "!";
};

// 3. Arrow Function (modern — use this most often)
const greet = (name) => {
  return "Hello, " + name + "!";
};

// Arrow shorthand (single expression — no return keyword needed)
const greet = (name) => "Hello, " + name + "!";
const double = n => n * 2; // single param — no parentheses needed

// Calling a function
console.log(greet("Myat")); // "Hello, Myat!"
console.log(double(5));     // 10

Parameters, Arguments, Return

Parameters are the variables listed in the function definition (the "placeholders"). Arguments are the actual values you pass in when calling. return sends a value back out. Without return, a function returns undefined.

// Parameters: price, taxRate
// Arguments: 100, 0.1
const calculateTotal = (price, taxRate) => {
  const tax = price * taxRate;
  return price + tax; // returns 110
};

const total = calculateTotal(100, 0.1);
console.log(total); // 110

// Default parameters (used if argument is missing)
const greet = (name = "Guest") => "Hello, " + name;
console.log(greet());        // "Hello, Guest"
console.log(greet("Myat"));  // "Hello, Myat"

Higher-Order Functions

Functions can be passed as arguments to other functions. This is a key concept in modern JS — you'll use it constantly with array methods and events.

// Passing a function as an argument
const runTwice = (fn) => {
  fn();
  fn();
};

const sayHi = () => console.log("Hi!");
runTwice(sayHi); // logs "Hi!" twice

// Anonymous function passed directly
runTwice(() => console.log("Hello!")); // logs "Hello!" twice

❌ Common Mistakes

  • Forgetting return — function runs but gives back undefined
  • Calling a function without parentheses — greet is the function itself, greet() calls it
  • Using this inside arrow functions when you need the object context — arrow functions don't have their own this
  • Writing a function that does too many things — one function, one job
  • Not naming functions clearly — doStuff() is bad, calculateCartTotal() is good

Quick Summary

  • Functions = reusable code blocks
  • Arrow functions are the modern standard
  • Parameters = placeholders, Arguments = real values
  • return sends the result back out
  • Default parameters handle missing arguments
  • Functions can be passed as values to other functions

Control Flow

Making decisions in your code — if, else, switch, ternary

if / else if / else

Control flow lets your code make decisions. Think of it like a flowchart — "if this is true, do this; otherwise, do that." This is how ALL programs work under the hood.

const score = 75;

if (score >= 90) {
  console.log("A — Excellent!");
} else if (score >= 80) {
  console.log("B — Good!");
} else if (score >= 70) {
  console.log("C — Average");
} else if (score >= 60) {
  console.log("D — Below average");
} else {
  console.log("F — Try again");
}
// logs "C — Average"

Ternary Operator

A one-line shortcut for if/else. Format: condition ? valueIfTrue : valueIfFalse. Great for simple yes/no decisions. Don't nest them — it gets unreadable fast.

const age = 20;

// Long way
let status;
if (age >= 18) {
  status = "Adult";
} else {
  status = "Minor";
}

// Ternary — one line
const status = age >= 18 ? "Adult" : "Minor";
console.log(status); // "Adult"

// In JSX / template strings
const message = `You are ${age >= 18 ? "an adult" : "a minor"}`;
console.log(message);

switch Statement

Better than long if/else chains when you're comparing one variable against many specific values. Always add break or execution "falls through" to the next case.

const day = "Monday";

switch (day) {
  case "Monday":
  case "Tuesday":
  case "Wednesday":
  case "Thursday":
  case "Friday":
    console.log("Weekday");
    break;
  case "Saturday":
  case "Sunday":
    console.log("Weekend!");
    break;
  default:
    console.log("Unknown day");
}

Short-Circuit Evaluation

JavaScript stops evaluating && as soon as it finds a false, and stops || as soon as it finds a true. This is used as a shorthand for conditionals.

const user = { name: "Myat" };

// && short-circuit: only runs if user exists
user && console.log(user.name); // "Myat"

// || short-circuit: use default if value is falsy
const name = user.name || "Guest"; // "Myat"
const points = 0 || 100;           // 100 (because 0 is falsy!)

// ?? nullish coalescing — only falls back for null/undefined
const points = 0 ?? 100;           // 0 (better for numbers!)

❌ Common Mistakes

  • Forgetting break in switch — causes fall-through to next case
  • Using || for default values when 0 or empty string is valid — use ?? instead
  • Nesting ternaries — hard to read, just use if/else
  • Using assignment = instead of comparison === inside if conditions
  • Forgetting that strings, objects, and arrays are always truthy even if "empty"

Quick Summary

  • if/else if/else for complex decisions
  • Ternary ? : for simple one-line if/else
  • switch when comparing one value to many
  • && short-circuits on false, || on true
  • ?? only falls back for null/undefined (use for defaults)
  • Objects and arrays are always truthy

Loops

Repeat actions without repeating code

Why Loops?

Imagine displaying 100 products on a page. You don't write 100 lines — you write the logic once and loop over the data. Loops are how every real app handles lists, tables, cards, and repeated actions.

// for loop — when you know how many times to run
for (let i = 0; i < 5; i++) {
  console.log("Item " + i);
}
// Item 0, Item 1, Item 2, Item 3, Item 4

// while loop — keep going until condition is false
let count = 0;
while (count < 3) {
  console.log("Count: " + count);
  count++;
}

// for...of — loop over arrays (modern, clean)
const fruits = ["apple", "banana", "cherry"];
for (const fruit of fruits) {
  console.log(fruit);
}

// for...in — loop over object keys
const user = { name: "Myat", age: 19, city: "Yangon" };
for (const key in user) {
  console.log(key + ": " + user[key]);
}

forEach — Array Loops

forEach is the most common way to loop through arrays in modern JS. It takes a callback function and runs it once for each item.

const products = ["Shoes", "Shirt", "Hat"];

// forEach with arrow function
products.forEach((product, index) => {
  console.log(`${index + 1}. ${product}`);
});
// 1. Shoes
// 2. Shirt
// 3. Hat

// Creating HTML cards from data
const names = ["Alice", "Bob", "Charlie"];
names.forEach(name => {
  const card = document.createElement("div");
  card.textContent = name;
  document.body.appendChild(card);
});

break and continue

break stops the loop completely. continue skips the current iteration and goes to the next one.

// break — stop when found
const nums = [1, 3, 5, 7, 2, 9];
for (const n of nums) {
  if (n % 2 === 0) {
    console.log("First even number: " + n);
    break; // stop looking
  }
}

// continue — skip odd numbers
for (let i = 0; i < 10; i++) {
  if (i % 2 !== 0) continue; // skip odds
  console.log(i); // 0, 2, 4, 6, 8
}

❌ Common Mistakes

  • Infinite loop — forgetting to increment i or never making the condition false — freezes the browser
  • Using for...in on arrays — use for...of or forEach for arrays instead
  • Modifying an array while looping over it — causes skipped items or errors
  • Off-by-one errors — i < arr.length vs i <= arr.length (the second goes one too far)
  • Using a regular for loop when forEach or array methods would be cleaner

Quick Summary

  • for — when you know the count
  • while — when condition-driven
  • for...of — arrays and iterables
  • forEach — array callback (most used)
  • for...in — object keys
  • break stops, continue skips

Arrays & Array Methods

Lists of data — the most important data structure in frontend JS

What is an Array?

An array is an ordered list of values. In real frontend apps, almost all your data comes as arrays — a list of products, users, posts, messages. Mastering array methods is the single biggest skill jump a beginner can make.

// Creating arrays
const colors = ["red", "green", "blue"];
const numbers = [1, 2, 3, 4, 5];
const mixed = ["Myat", 19, true, null]; // can mix types

// Accessing items (zero-indexed)
console.log(colors[0]); // "red"
console.log(colors[2]); // "blue"
console.log(colors[colors.length - 1]); // last item: "blue"

// Modifying
colors.push("yellow");     // add to end
colors.pop();              // remove from end
colors.unshift("purple");  // add to start
colors.shift();            // remove from start
colors[1] = "orange";      // replace item at index 1

The Big 3 — map, filter, reduce

These three methods are used in every single frontend project. Learn these and you can handle almost any data transformation task.

const products = [
  { name: "Shoes",  price: 59.99, inStock: true  },
  { name: "Shirt",  price: 29.99, inStock: false },
  { name: "Hat",    price: 19.99, inStock: true  },
  { name: "Jacket", price: 89.99, inStock: true  },
];

// map — transform every item, returns NEW array (same length)
const names = products.map(p => p.name);
// ["Shoes", "Shirt", "Hat", "Jacket"]

const prices = products.map(p => p.price * 1.1); // 10% markup
// [65.99, 32.99, 21.99, 98.99]

// filter — keep only items that pass the test, returns NEW array
const available = products.filter(p => p.inStock);
// [{ Shoes }, { Hat }, { Jacket }]

const affordable = products.filter(p => p.price < 50);
// [{ Shirt }, { Hat }]

// reduce — boil an array down to a SINGLE value
const total = products.reduce((sum, p) => sum + p.price, 0);
// 199.96  (starting at 0, adds each price)

Other Essential Methods

const nums = [3, 1, 4, 1, 5, 9, 2, 6];

// find — first item that matches
const firstBig = nums.find(n => n > 4); // 5

// findIndex — index of first match
const idx = nums.findIndex(n => n > 4); // 4

// some — does ANY item match?
const hasNegative = nums.some(n => n < 0); // false

// every — do ALL items match?
const allPositive = nums.every(n => n > 0); // true

// includes — is this value in the array?
nums.includes(9); // true

// sort — sorts IN PLACE (careful!)
const sorted = [...nums].sort((a, b) => a - b); // ascending

// flat / flatMap
const nested = [[1, 2], [3, 4], [5]];
nested.flat(); // [1, 2, 3, 4, 5]

// slice — copy a portion (non-destructive)
nums.slice(1, 4); // [1, 4, 1] (index 1 to 3)

// splice — remove/insert items (destructive — modifies original!)
nums.splice(2, 1); // removes 1 item at index 2

❌ Common Mistakes

  • Thinking map modifies the original — it returns a NEW array, original is unchanged
  • Forgetting the initial value in reduce — crashes on empty arrays
  • Using sort() without a compare function — sorts alphabetically even for numbers ([10, 9, 2] stays wrong)
  • Confusing slice (non-destructive copy) with splice (modifies original)
  • Mutating an array that was passed into a function — use spread [...arr] to make a copy first

Quick Summary

  • map — transform every item → new array
  • filter — keep items that pass → new array
  • reduce — collapse to single value
  • find / findIndex — find first match
  • some / every — check conditions
  • push/pop/shift/unshift — add/remove items

Objects & Destructuring

Key-value data — how real-world data is structured

What is an Object?

An object stores data as key-value pairs. Almost all real data in JS is an object — a user, a product, an API response. Think of it like a form with labeled fields.

// Creating an object
const user = {
  firstName: "Myat",
  lastName: "Kyaw",
  age: 19,
  isLoggedIn: true,
  address: {         // nested object
    city: "Yangon",
    country: "Myanmar"
  },
  greet() {          // method (function inside object)
    return `Hello, I'm ${this.firstName}`;
  }
};

// Accessing values
console.log(user.firstName);           // "Myat"  (dot notation)
console.log(user["lastName"]);         // "Kyaw"  (bracket notation)
console.log(user.address.city);        // "Yangon"
console.log(user.greet());             // "Hello, I'm Myat"

// Updating values
user.age = 20;
user.email = "myat@email.com"; // add new key

// Deleting
delete user.isLoggedIn;

Destructuring — Extract Values Cleanly

Destructuring lets you pull values out of objects and arrays into separate variables in one line. This is used constantly in modern React and API calls.

const user = { name: "Myat", age: 19, city: "Yangon" };

// Without destructuring (old way)
const name = user.name;
const age = user.age;

// With destructuring (modern way)
const { name, age, city } = user;
console.log(name); // "Myat"

// Rename while destructuring
const { name: fullName, age: years } = user;
console.log(fullName); // "Myat"

// Default values
const { name, role = "user" } = user;
console.log(role); // "user" (key didn't exist, used default)

// Nested destructuring
const { address: { city } } = user; // only works if address exists

// Array destructuring
const rgb = [255, 128, 0];
const [red, green, blue] = rgb;
console.log(red); // 255

// Swap variables with array destructuring
let a = 1, b = 2;
[a, b] = [b, a];
console.log(a, b); // 2, 1

Spread & Rest with Objects

The spread operator ... copies properties into a new object. Rest collects remaining properties.

const defaults = { theme: "dark", lang: "en", fontSize: 16 };
const userPrefs = { theme: "light", fontSize: 18 };

// Merge objects (later keys overwrite earlier ones)
const settings = { ...defaults, ...userPrefs };
// { theme: "light", lang: "en", fontSize: 18 }

// Copy object (avoid mutating original)
const original = { name: "Myat", age: 19 };
const copy = { ...original, age: 20 }; // new object with updated age

// Rest — collect remaining properties
const { name, ...rest } = user;
console.log(rest); // { age: 19, city: "Yangon" }

❌ Common Mistakes

  • Directly copying an object with = — both variables point to the same object in memory, mutating one mutates both. Use { ...obj }
  • Accessing a property of null or undefined — crashes with "Cannot read properties of undefined". Use optional chaining: user?.address?.city
  • Forgetting that { ...obj } is a shallow copy — nested objects are still shared
  • Using for...in on objects when you only want the object's own properties — use Object.keys()

Quick Summary

  • Objects = key-value pairs
  • Dot notation obj.key or bracket obj["key"]
  • Destructuring extracts values into variables
  • Spread ... copies/merges objects
  • Optional chaining ?. prevents null crashes
  • Use Object.keys() / Object.values() to iterate

DOM Manipulation

Reading and changing the webpage with JavaScript

What is the DOM?

The DOM (Document Object Model) is JavaScript's representation of your HTML. When you load a page, the browser builds a tree of objects — every tag becomes a node. JS can read, add, remove, or change any part of that tree to update what the user sees without reloading.

// Selecting elements
const title     = document.getElementById("title");
const btn       = document.querySelector(".submit-btn");   // CSS selector
const allCards  = document.querySelectorAll(".card");       // NodeList
const inputs    = document.querySelectorAll("input[type='text']");

// Reading values
console.log(title.textContent);   // text only
console.log(title.innerHTML);     // including HTML tags
console.log(btn.getAttribute("data-id")); // custom attribute

// Changing content
title.textContent = "New Title";
btn.innerHTML = "Click Me";

// Changing styles
title.style.color = "red";
title.style.fontSize = "24px";

// Classes (best practice — use classList, not .style)
title.classList.add("active");
title.classList.remove("hidden");
title.classList.toggle("dark-mode"); // add if missing, remove if present
title.classList.contains("active"); // true/false

Creating & Inserting Elements

You can build HTML elements from scratch in JS and insert them anywhere on the page. This is how every todo list, product card, and notification gets added dynamically.

// Create a new element
const card = document.createElement("div");
card.classList.add("card");
card.textContent = "New Card";

// Add to the page
document.body.appendChild(card);           // add to end of body
document.body.prepend(card);               // add to start of body

const container = document.querySelector(".container");
container.appendChild(card);              // inside container

// Insert at specific position
container.insertAdjacentHTML("beforeend", `
  <div class="card">
    <h3>Dynamic Title</h3>
    <p>Dynamic content</p>
  </div>
`);

// Remove an element
card.remove();
// or: container.removeChild(card);

Real World: Render a List from Data

const products = ["Shoes", "Shirt", "Hat", "Jacket"];
const list = document.querySelector("#product-list");

// Clear first, then rebuild
list.innerHTML = "";

products.forEach(product => {
  const li = document.createElement("li");
  li.textContent = product;
  list.appendChild(li);
});

// Cleaner with map + join
list.innerHTML = products
  .map(p => `<li class="item">${p}</li>`)
  .join("");

❌ Common Mistakes

  • Running DOM code before the page loads — put your <script> at the bottom of body, or use DOMContentLoaded
  • Using innerHTML with user input — security risk (XSS). Use textContent for user data
  • Selecting an element that doesn't exist — querySelector returns null and then crashes on the next line
  • Forgetting that querySelectorAll returns a NodeList, not an array — use Array.from() or spread to use array methods on it
  • Using .style.display = "none" everywhere — better to toggle a CSS class

Quick Summary

  • querySelector — select one element
  • querySelectorAll — select all matches
  • textContent — safe text, innerHTML — HTML string
  • classList.add/remove/toggle — best practice for visual changes
  • createElement + appendChild — dynamic elements
  • insertAdjacentHTML — insert HTML string at position

Events

Responding to user actions — clicks, typing, scrolling, submitting

addEventListener

Events are how your page becomes interactive. The user does something (click, type, scroll) and your JS responds. You attach event listeners to elements and provide a callback function to run when the event fires.

const btn = document.querySelector("#my-btn");

// Basic click event
btn.addEventListener("click", () => {
  console.log("Button was clicked!");
});

// The event object — contains info about the event
btn.addEventListener("click", (event) => {
  console.log(event.target);        // the element clicked
  console.log(event.type);          // "click"
  event.preventDefault();           // stop default browser behavior
});

// Common events
element.addEventListener("click",      handler); // click
element.addEventListener("mouseover",  handler); // hover
element.addEventListener("mouseout",   handler); // leave hover
element.addEventListener("keydown",    handler); // key pressed
element.addEventListener("keyup",      handler); // key released
element.addEventListener("input",      handler); // typing in input
element.addEventListener("change",     handler); // value changed
element.addEventListener("submit",     handler); // form submitted
element.addEventListener("scroll",     handler); // page scrolled
window.addEventListener("load",        handler); // page fully loaded
document.addEventListener("DOMContentLoaded", handler); // HTML parsed

Event Delegation — The Pro Technique

Instead of adding a listener to every item (bad for performance), add one listener to the parent and check which child was clicked. Essential for dynamic lists where items are added/removed.

// BAD — listener on every button (imagine 1000 items)
document.querySelectorAll(".delete-btn").forEach(btn => {
  btn.addEventListener("click", () => { /* delete logic */ });
});

// GOOD — one listener on the parent
const list = document.querySelector("#todo-list");
list.addEventListener("click", (event) => {
  // Check if the clicked thing was a delete button
  if (event.target.classList.contains("delete-btn")) {
    event.target.closest("li").remove();
  }
  // Check for other buttons
  if (event.target.classList.contains("complete-btn")) {
    event.target.closest("li").classList.toggle("done");
  }
});

Form Events

Forms are how users send data. Always call event.preventDefault() on submit to stop the page from reloading.

const form = document.querySelector("#signup-form");

form.addEventListener("submit", (event) => {
  event.preventDefault(); // stop page reload!

  const nameInput  = document.querySelector("#name");
  const emailInput = document.querySelector("#email");

  const data = {
    name:  nameInput.value.trim(),
    email: emailInput.value.trim()
  };

  if (!data.name || !data.email) {
    alert("Please fill in all fields");
    return;
  }

  console.log("Form data:", data);
  form.reset(); // clear the form
});

// Live input feedback
const nameInput = document.querySelector("#name");
nameInput.addEventListener("input", (e) => {
  console.log("Typing:", e.target.value);
});

❌ Common Mistakes

  • Forgetting event.preventDefault() on form submit — page reloads and you lose data
  • Adding event listeners inside a loop — creates duplicate listeners
  • Using inline onclick="" in HTML — mixing JS into HTML, hard to manage
  • Not removing event listeners when no longer needed — causes memory leaks
  • Forgetting that event.target is the actual element clicked (could be a child), not always the element you attached the listener to

Quick Summary

  • addEventListener(type, callback) is the standard way
  • event.target = what was actually clicked
  • event.preventDefault() stops default behavior
  • Event delegation = one listener on parent (performance)
  • Always preventDefault() on form submit
  • .value.trim() to read and clean input fields

ES6+ Modern Syntax

The modern features that make JavaScript clean and powerful

Template Literals

Backtick strings that let you embed variables and expressions directly. No more messy string concatenation.

const name = "Myat";
const age  = 19;

// Old way (ugly)
const msg = "Hello, " + name + "! You are " + age + " years old.";

// Template literal (clean)
const msg = `Hello, ${name}! You are ${age} years old.`;

// Expressions inside
const price = 99.9;
const tax   = 0.1;
console.log(`Total: $${(price * (1 + tax)).toFixed(2)}`); // Total: $109.89

// Multi-line strings
const html = `
  <div class="card">
    <h2>${name}</h2>
    <p>Age: ${age}</p>
  </div>
`;

Spread & Rest Operator

... does two opposite things: spread expands an array/object into individual items, rest collects multiple items into one array.

// SPREAD — expand
const arr1 = [1, 2, 3];
const arr2 = [4, 5, 6];
const combined = [...arr1, ...arr2]; // [1, 2, 3, 4, 5, 6]

// Copy array (not the same reference!)
const copy = [...arr1];

// Pass array items as separate arguments
Math.max(...arr1); // same as Math.max(1, 2, 3)

// REST — collect remaining items into array
const [first, second, ...others] = [1, 2, 3, 4, 5];
console.log(others); // [3, 4, 5]

// Rest in function parameters
const sum = (...numbers) => numbers.reduce((a, b) => a + b, 0);
sum(1, 2, 3, 4, 5); // 15

Optional Chaining & Nullish Coalescing

These two operators prevent crashes when dealing with data that might not exist — extremely common when working with API data.

const user = {
  name: "Myat",
  address: {
    city: "Yangon"
  }
};

// Without optional chaining — CRASHES if address is undefined
console.log(user.address.city);      // "Yangon"
console.log(user.profile.avatar);    // TypeError: Cannot read...

// With optional chaining — returns undefined instead of crashing
console.log(user?.profile?.avatar);  // undefined (safe!)
console.log(user?.address?.city);    // "Yangon"

// Optional method calls
user.greet?.();     // only calls if greet exists
arr?.map(fn);       // only maps if arr exists

// Nullish coalescing ?? — fallback ONLY for null/undefined
const username = user?.username ?? "Guest";  // "Guest"
const count    = 0 ?? 10;                    // 0 (not 10! 0 is valid)

Short-Hand Object Properties & Computed Keys

const name = "Myat";
const age  = 19;

// Short-hand: if key and variable name are the same
const user = { name, age }; // same as { name: name, age: age }

// Computed property keys
const field = "email";
const data = {
  [field]: "myat@email.com" // key is the VALUE of field
};
// { email: "myat@email.com" }

❌ Common Mistakes

  • Forgetting backticks for template literals and using quotes instead — "Hello ${name}" outputs literally Hello ${name}
  • Spreading an object into an array or vice versa — [...obj] throws an error unless the object is iterable
  • Using || for default values when 0 or "" are valid — use ??
  • Chaining too many optional chains — a?.b?.c?.d?.e hides bugs, sometimes you WANT it to crash so you know data is wrong

Quick Summary

  • Template literals use backticks and ${}
  • Spread ... expands, Rest ... collects
  • Optional chaining ?. prevents null crashes
  • ?? fallback only for null/undefined
  • Short-hand property: { name } = { name: name }
  • These are used in every single modern JS codebase

Fetch API & Async/Await

Getting real data from the internet — the most important skill for modern frontend

What is an API?

An API (Application Programming Interface) is a URL you can request data from. The data comes back as JSON (JavaScript Object Notation) — which looks exactly like a JS object. Every real website uses APIs — weather apps, Twitter, YouTube, banking apps.

Promises — JavaScript's Async System

When you fetch data from the internet, JS doesn't wait for it — it continues running other code. A Promise is a placeholder that says "I'll have the value eventually — here's what to do when it arrives." async/await is the clean way to work with Promises.

// async/await — the modern clean way
const getUsers = async () => {
  try {
    // fetch returns a Promise — await pauses until it resolves
    const response = await fetch("https://jsonplaceholder.typicode.com/users");

    // response.json() also returns a Promise — await again
    const users = await response.json();

    console.log(users); // array of user objects
    return users;

  } catch (error) {
    // runs if fetch fails (network down, 404, etc.)
    console.error("Error fetching users:", error);
  }
};

// Calling an async function
getUsers(); // returns a Promise
getUsers().then(users => console.log(users)); // use .then() if needed

Checking the Response Status

A fetch only throws an error on network failure. A 404 or 500 response is considered "successful" by fetch — you must check manually.

const getPost = async (id) => {
  try {
    const response = await fetch(`https://jsonplaceholder.typicode.com/posts/${id}`);

    // Check HTTP status
    if (!response.ok) {
      throw new Error(`HTTP error! Status: ${response.status}`);
    }

    const post = await response.json();
    return post;

  } catch (error) {
    console.error("Failed:", error.message);
    return null;
  }
};

Sending Data — POST Request

Fetch can also SEND data to a server (creating a new user, submitting a form).

const createPost = async (postData) => {
  try {
    const response = await fetch("https://jsonplaceholder.typicode.com/posts", {
      method: "POST",
      headers: {
        "Content-Type": "application/json"
      },
      body: JSON.stringify(postData) // convert object to JSON string
    });

    const newPost = await response.json();
    console.log("Created:", newPost);
    return newPost;

  } catch (error) {
    console.error("Failed to create post:", error);
  }
};

createPost({ title: "My Post", body: "Hello world", userId: 1 });

Render API Data to the DOM

const container = document.querySelector("#users");

const displayUsers = async () => {
  container.innerHTML = "<p>Loading...</p>";

  const response = await fetch("https://jsonplaceholder.typicode.com/users");
  const users    = await response.json();

  container.innerHTML = users
    .map(user => `
      <div class="card">
        <h3>${user.name}</h3>
        <p>${user.email}</p>
        <p>${user.address.city}</p>
      </div>
    `)
    .join("");
};

displayUsers();

❌ Common Mistakes

  • Forgetting await — you get a Promise object, not the data
  • Not wrapping in try/catch — unhandled promise rejections crash silently
  • Not checking response.ok — a 404 won't throw automatically
  • Forgetting JSON.stringify() when sending — sending a raw object in body does nothing
  • Using async in a forEach loop — the loop won't wait for each call. Use for...of with await instead

Quick Summary

  • fetch(url) returns a Promise
  • await pauses until Promise resolves
  • async keyword required to use await
  • Always check response.ok
  • response.json() parses the JSON
  • Wrap everything in try/catch

Error Handling

Writing code that doesn't crash when things go wrong

try / catch / finally

try — run this code. catch — if it throws an error, do this instead. finally — always run this, error or not (used for cleanup like hiding loading spinners).

const parseUserData = (jsonString) => {
  try {
    const data = JSON.parse(jsonString); // might throw SyntaxError
    return data;
  } catch (error) {
    console.error("Invalid JSON:", error.message);
    return null; // return safe fallback
  } finally {
    console.log("parseUserData finished"); // always runs
  }
};

parseUserData('{"name":"Myat"}');  // works
parseUserData("not valid json");   // caught, returns null

throw — Create Your Own Errors

You can throw errors manually to signal that something went wrong in your own code — like invalid inputs.

const divide = (a, b) => {
  if (b === 0) {
    throw new Error("Cannot divide by zero");
  }
  return a / b;
};

try {
  console.log(divide(10, 2));  // 5
  console.log(divide(10, 0));  // throws error
} catch (error) {
  console.error(error.message); // "Cannot divide by zero"
}

// Custom error types
class ValidationError extends Error {
  constructor(message) {
    super(message);
    this.name = "ValidationError";
  }
}

const validateAge = (age) => {
  if (typeof age !== "number") throw new ValidationError("Age must be a number");
  if (age < 0 || age > 150)   throw new ValidationError("Age out of valid range");
  return true;
};

Error Types to Know

Error TypeWhen it happens
TypeErrorWrong type — calling non-function, accessing null property
ReferenceErrorUsing a variable that doesn't exist
SyntaxErrorInvalid JS or invalid JSON.parse input
RangeErrorNumber out of valid range
NetworkErrorFetch failed (no internet, CORS)

❌ Common Mistakes

  • Empty catch blocks — swallowing errors silently makes debugging a nightmare
  • Catching too broadly — catching everything hides bugs you should fix
  • Not returning a fallback value from catch — the caller gets undefined and crashes later
  • Forgetting async errors need try/catch too — or .catch() on the promise

Quick Summary

  • try/catch prevents crashes
  • finally always runs (cleanup)
  • throw new Error() creates custom errors
  • Always log error.message in catch
  • Return safe fallback values from catch
  • Know the 5 main error types

Scope & Closures

Where variables live — the concept that explains half of all JS bugs

Scope — Where Can You Access a Variable?

Scope is the region where a variable exists. JS has three kinds: global (everywhere), function (inside a function), and block (inside curly braces with let/const).

// Global scope — accessible everywhere
const appName = "MyApp";

const greetUser = (name) => {
  // Function scope — only inside greetUser
  const greeting = "Hello";

  if (name.length > 3) {
    // Block scope — only inside this if block (with let/const)
    const message = `${greeting}, ${name}!`;
    console.log(message); // works
  }

  // console.log(message); // ReferenceError! message doesn't exist here
  console.log(greeting);  // works — greeting is in function scope
};

// console.log(greeting); // ReferenceError! greeting is inside function

Closures — Functions That Remember

A closure is when a function remembers variables from its outer scope even after that scope has finished running. This sounds complex but you use it all the time — event handlers, counters, module patterns.

// Closure — makeCounter returns a function that closes over count
const makeCounter = () => {
  let count = 0; // this variable is "closed over"

  return {
    increment: () => ++count,
    decrement: () => --count,
    getCount:  () => count
  };
};

const counter = makeCounter();
counter.increment(); // 1
counter.increment(); // 2
counter.increment(); // 3
counter.decrement(); // 2
console.log(counter.getCount()); // 2

// Each call creates its own independent closure
const counter2 = makeCounter();
counter2.increment(); // 1 — separate from counter!

Real World — Closure for Private Data

// Closure to create a shopping cart
const createCart = () => {
  const items = []; // private — can't access from outside

  return {
    add:   (item) => items.push(item),
    remove:(item) => items.splice(items.indexOf(item), 1),
    total: ()     => items.reduce((sum, i) => sum + i.price, 0),
    list:  ()     => [...items] // return copy, not the real array
  };
};

const cart = createCart();
cart.add({ name: "Shoes", price: 59.99 });
cart.add({ name: "Hat",   price: 19.99 });
console.log(cart.total()); // 79.98
// cart.items — undefined! private to the closure

❌ Common Mistakes

  • The classic loop closure bug — using var in a for loop and expecting each iteration to capture its own value. Use let instead
  • Accidentally creating global variables by forgetting const or let — the variable leaks to global scope
  • Memory leaks — closures hold references to outer scope, keeping large objects in memory

Quick Summary

  • Scope = where a variable can be accessed
  • Global → function → block (narrowest wins)
  • Closure = function remembers outer variables
  • Used for: counters, private data, event handlers
  • let/const are block-scoped, var is function-scoped
  • Each closure call creates independent state

Local Storage

Save data in the browser — persists after page refresh

What is Local Storage?

Local Storage is a mini database in the browser. Data saved here persists even after closing the tab or restarting the browser — until you clear it or the user clears their browser data. Real world uses: saved theme (dark/light), shopping cart, last searched items, user preferences.

// Save — key/value pairs (values must be strings)
localStorage.setItem("theme", "dark");
localStorage.setItem("username", "Myat");

// Read
const theme    = localStorage.getItem("theme");    // "dark"
const username = localStorage.getItem("username"); // "Myat"
const missing  = localStorage.getItem("xyz");      // null

// Remove one item
localStorage.removeItem("username");

// Remove everything
localStorage.clear();

Saving Objects & Arrays — JSON

Local Storage only stores strings. To save objects or arrays, convert to JSON string first with JSON.stringify(), and parse it back with JSON.parse().

// Saving an object
const user = { name: "Myat", theme: "dark", fontSize: 16 };
localStorage.setItem("user", JSON.stringify(user));

// Reading it back
const saved = JSON.parse(localStorage.getItem("user"));
console.log(saved.name); // "Myat"

// Saving an array (like a todo list)
const todos = ["Buy milk", "Read docs", "Build project"];
localStorage.setItem("todos", JSON.stringify(todos));

const loadedTodos = JSON.parse(localStorage.getItem("todos")) || [];

// Full pattern: load → use → save
const getTodos = () =>
  JSON.parse(localStorage.getItem("todos")) || [];

const saveTodos = (todos) =>
  localStorage.setItem("todos", JSON.stringify(todos));

// Usage
const todos = getTodos();
todos.push("New item");
saveTodos(todos);

Dark Mode Toggle — Real World Example

// On page load — apply saved theme
const savedTheme = localStorage.getItem("theme") || "light";
document.body.classList.add(savedTheme);

// Toggle button
const themeBtn = document.querySelector("#theme-toggle");
themeBtn.addEventListener("click", () => {
  const isDark = document.body.classList.toggle("dark");
  localStorage.setItem("theme", isDark ? "dark" : "light");
});

❌ Common Mistakes

  • Storing objects without JSON.stringify() — saves as "[object Object]" instead of real data
  • Forgetting JSON.parse() when reading — gives you a string instead of an object
  • Not handling null from getItem() — crashes when trying to JSON.parse(null). Use || [] or || {}
  • Storing sensitive data (passwords, tokens) in Local Storage — it's readable by any JS on the page
  • Treating Local Storage like a real database — it's only for small UI preferences, not real app data

Quick Summary

  • Stores data across page refreshes
  • setItem / getItem / removeItem
  • Only stores strings — use JSON.stringify/parse
  • Always handle null from getItem
  • Never store passwords or tokens
  • ~5MB limit per domain

Modules (import / export)

Split your code into organized files — how every real project is structured

Why Modules?

As your project grows, one giant JS file becomes unmanageable. Modules let you split code into separate files, each responsible for one thing. Then import only what you need where you need it. This is how every React, Vue, and Node.js project is structured.

// ── utils.js ── (a utility file)

// Named export — export multiple things
export const formatPrice  = (price) => `$${price.toFixed(2)}`;
export const capitalize   = (str)   => str[0].toUpperCase() + str.slice(1);
export const clamp        = (n, min, max) => Math.min(Math.max(n, min), max);

// Default export — one main thing per file
export default class Cart {
  constructor() { this.items = []; }
  add(item)  { this.items.push(item); }
  total()    { return this.items.reduce((s, i) => s + i.price, 0); }
}
// ── main.js ── (your main file)

// Import named exports — use curly braces
import { formatPrice, capitalize } from "./utils.js";

// Import default export — any name, no braces
import Cart from "./utils.js";

// Import everything as a namespace
import * as utils from "./utils.js";
utils.formatPrice(9.99); // "$9.99"

// Using the imports
const price = formatPrice(29.99);  // "$29.99"
const name  = capitalize("myat");  // "Myat"

const cart = new Cart();
cart.add({ name: "Shoes", price: 59.99 });
console.log(cart.total()); // 59.99

How to Use Modules in HTML

Add type="module" to your script tag. Note: modules don't work with the file:// protocol — you need a local server (VS Code's Live Server extension works).

<!-- In your HTML -->
<script type="module" src="main.js"></script>

<!-- Modules are deferred by default (runs after HTML loads) -->
<!-- Each module has its own scope — no global leaks -->

Real Project Structure with Modules

project/
├── index.html
├── js/
│   ├── main.js          // entry point
│   ├── api.js           // all fetch calls
│   ├── ui.js            // DOM manipulation functions
│   ├── storage.js       // localStorage helpers
│   └── utils.js         // pure utility functions

// api.js
export const fetchUsers = async () => {
  const res = await fetch("/api/users");
  return res.json();
};

// ui.js
export const renderUsers = (users, container) => {
  container.innerHTML = users.map(u => `
    <div class="card">${u.name}</div>
  `).join("");
};

// main.js

import { renderUsers } from "./ui.js";

  const users = await fetchUsers();
  renderUsers(users, document.querySelector("#app"));

❌ Common Mistakes

  • Forgetting type="module" in the script tag — imports/exports won't work
  • Opening HTML directly with file:// — modules need a local server due to CORS. Use VS Code Live Server
  • Mixing default and named exports confusingly — pick one pattern per file and be consistent
  • Circular imports — file A imports from B which imports from A — causes hard bugs
  • Not including the .js extension in import paths — required in native modules

Quick Summary

  • Modules split code into organized files
  • Named export: export const fn, import: { fn }
  • Default export: export default, import: any name
  • Add type="module" to script tag
  • Modules need a local server (not file://)
  • One responsibility per file — the key to clean code
