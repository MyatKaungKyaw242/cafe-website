Wait — What Even IS HTML?

HTML stands for HyperText Markup Language. That sounds complicated, but it just means: a language that uses special labels (called tags) to tell the browser what each piece of content IS.

You write HTML in a plain text file saved as .html. You open that file in a browser (Chrome, Edge, Firefox). The browser reads your tags and displays the content correctly. That's it. That's a webpage.

HTML is NOT a programming language — it has no logic, no calculations, no decisions. It's purely a way to describe and organise content. Think of it like writing a letter where you also label every part: "this is the title", "this is a paragraph", "this is a list".

🏗️
HTML

The walls, floors, and rooms of a building. The structure and layout — what exists and where.

🎨
CSS

The paint, furniture, and decoration. Makes the structure look beautiful. You'll learn this next.

⚡
JavaScript

The electricity, doors, and elevators. Makes things interactive and dynamic. Comes after CSS.

Two words you'll see everywhere:   Semantic = meaningful / descriptive (a tag that says WHAT the content is).   Screen reader = software that reads webpages aloud for blind users.   SEO = how well Google can find and understand your page.

Everything You Need to Know — HTML Cheat Sheet

Page Structure

Think of this as the skeleton every HTML file must have. Without it, your page may look broken or behave differently across browsers. Every single .html file you ever create starts with exactly this.

<!DOCTYPE html> <html lang="en"> <head> <meta charset="UTF-8"> <meta name="viewport" content="width=device-width"> <title>Page Title</title> </head> <body> ...content here... </body> </html>

Headings & Text

Headings are like chapter titles in a book — h1 is the main title (only one per page!), h2 is a section title, and they get smaller down to h6. p is for normal paragraph text. strong makes text bold, em makes it italic.

<h1>Main Title (ONE per page)</h1> <h2>Section Heading</h2> <h3> ... <h6>Smallest</h6> <p>Normal paragraph text here.</p> <strong>Bold — important text</strong> <em>Italic — emphasis</em> <br> ← forces a new line <hr> ← draws a horizontal line <!-- Browser ignores this comment -->

Links & Images

a (anchor) creates a clickable link. href is where it goes — it's required. img puts a picture on the page. src is the path to the image file. alt is what shows if the image fails to load, and screen readers read it aloud — never skip it.

<a href="about.html">Go to About</a> <a href="https://google.com" target="_blank" rel="noopener noreferrer"> Opens safely in a new tab </a> <img src="photo.jpg" alt="A dog playing in the park" width="300"> <!-- alt="" only if image is decorative -->

Lists

ul is like a shopping list — bullets, order doesn't matter. ol is like a recipe — numbered steps, order matters. Every single item in both types is wrapped in an li tag. You can put a whole new list inside an li to make sub-items.

<!-- ul = unordered, bullets --> <ul> <li>HTML</li> <li>CSS</li> <li>JavaScript</li> </ul> <!-- ol = ordered, numbered steps --> <ol> <li>Open VS Code</li> <li>Create index.html</li> <li>Write your HTML</li> </ol>

Tables

Think of a table like a spreadsheet. thead holds the top header row (column names). tbody holds all the actual data rows. tr means "table row." th means "table header cell" (bold by default). td means "table data cell" (normal text).

<table> <thead> <tr> <th>Name</th> <!-- column title --> <th>Age</th> </tr> </thead> <tbody> <tr> <td>Alice</td> <!-- data cell --> <td>24</td> </tr> </tbody> </table>

Forms

Forms collect information from users. input is the box they type into — its type attribute changes what kind of data it accepts. label tells users what the field is for and must be connected to its input using for and id. button submits the form.

<form action="#" method="post"> <label for="nm">Your Name</label> <input type="text" id="nm" name="nm" required placeholder="Type here..."> <input type="email"> <!-- validates @ --> <input type="password"> <!-- hides text --> <input type="checkbox"> <!-- tick box --> <input type="radio"> <!-- pick one --> <input type="date"> <!-- date picker --> <select> <option>Choice A</option> <option>Choice B</option> </select> <textarea rows="4"></textarea> <button type="submit">Send</button> </form>

Semantic Tags

These tags have meaning built in. When Google reads your page, it understands that a <nav> is navigation and a <footer> is the bottom section. When a blind user's software reads your page, it knows to announce "navigation landmark" when it hits <nav>. A <div> tells them nothing at all. Use these tags instead.

<header> <!-- Logo + main navigation --> </header> <nav> <!-- navigation links --></nav> <main> <!-- ONE per page --> <section> <!-- grouped topic --></section> <article> <!-- standalone piece --></article> <aside> <!-- sidebar/extra info --></aside> </main> <footer> <!-- bottom of page --></footer>

Inline vs Block

Block elements always start on a new line and stretch the full width — like paragraphs. Inline elements sit inside text without breaking the flow — like bold words inside a sentence. You cannot put a block element inside an inline element.

<!-- BLOCK: starts on its own line --> <div> <p> <h1>–<h6> <ul> <ol> <li> <table> <header> <section> <main> etc. <!-- INLINE: stays in the text flow --> <span> <a> <strong> <em> <img> <input> <label> <button> <!-- WRONG — block inside inline: --> <span><p>This breaks HTML!</p></span>

Special Characters

Some characters have special meaning in HTML — like < and > which the browser thinks are tags. If you want to display them as actual text on screen, you must use these escape codes instead. They always start with & and end with ;.

&amp; → & (ampersand) &lt; → < (less than) &gt; → > (greater than) &nbsp; → · (non-breaking space) &copy; → © (copyright) &trade; → ™ (trademark) &mdash; → — (long dash) &hearts; → ♥ (heart)
How to Use — Elements & Attributes Deep Dive

ul / ol / li — Lists

<ul> = unordered (bullets)  |  <ol> = ordered (numbers)
Every item goes inside <li>. You can nest lists inside any <li>. 

<ul>
  <li>HTML</li>
  <li>CSS
    <ul>
      <li>Flexbox</li>
      <li>Grid</li>
    </ul>
  </li>
</ul>

<!-- Ordered with custom start -->
<ol type="A" start="3">
  <li>Starts at C</li>
  <li>Then D</li>
</ol>

type="1 / A / a / I / i" — ol only. Navigation menus are almost always ul > li > a.

a — The Anchor Tag (Links)

href is the destination. It can point to a page, a section, an email, a phone number, or a file to download. 

<!-- Same site (relative) -->
<a href="about.html">About</a>

<!-- External (new tab safely) -->
<a href="https://google.com"
   target="_blank"
   rel="noopener noreferrer">
  Google
</a>

<!-- Jump to section on same page -->
<a href="#contact">Contact</a>
<section id="contact">...</section>

<!-- Email / Phone -->
<a href="mailto:hi@example.com">Email</a>
<a href="tel:+1234567890">Call</a>

<!-- Download a file -->
<a href="resume.pdf" download>
  Download CV
</a>

Always add rel="noopener noreferrer" with target="_blank" — without it the new tab can access your page via window.opener.

nav — Navigation

<nav> is a semantic wrapper for navigation links. Screen readers announce it as a landmark so users can jump straight to it. A page can have more than one <nav>. 

<!-- Main site nav -->
<nav>
  <ul>
    <li><a href="index.html">Home</a></li>
    <li><a href="about.html">About</a></li>
    <li><a href="contact.html">Contact</a></li>
  </ul>
</nav>

<!-- Footer nav (second nav is fine) -->
<footer>
  <nav aria-label="Footer links">
    <a href="privacy.html">Privacy</a>
    <a href="terms.html">Terms</a>
  </nav>
</footer>

Use aria-label="Footer links" when you have multiple navs so screen readers can tell them apart.

section — Grouped Content

<section> groups related content that belongs together. It should always have a heading inside. Not to be confused with <article> or <div>. 

<main>
  <section id="about">
    <h2>About Me</h2>
    <p>I build websites...</p>
  </section>

  <section id="skills">
    <h2>Skills</h2>
    <ul>...</ul>
  </section>
</main>

section = part of a larger page.  |  article = fully standalone (blog post, comment, product card).  |  div = no meaning at all — last resort only.

id — Unique Identifier

Every id must be unique on the page — only one element can have a given id. Used for anchor links, CSS targeting, and JavaScript. 

<!-- Anchor link jumps here -->
<section id="contact">
  <h2>Contact</h2>
</section>

<!-- Link to it from anywhere -->
<a href="#contact">Go to Contact</a>

<!-- Works across pages too -->
<a href="about.html#team">Meet the Team</a>

<!-- CSS (use sparingly) -->
<style>
  #contact { background: #111; }
</style>

Never duplicate an id. If you need to style multiple elements the same way, use class instead.

class — Reusable Label

class can be reused on as many elements as you want. One element can also have multiple classes separated by spaces. 

<!-- Same class on many elements -->
<div class="card">...</div>
<div class="card">...</div>
<div class="card">...</div>

<!-- Multiple classes on one element -->
<div class="card featured large">
  ...
</div>

<!-- CSS targets all .card elements -->
<style>
  .card { border: 1px solid #ccc; }
  .featured { background: gold; }
</style>

Use id when you mean "this one specific element." Use class when you mean "all elements of this type."

div & span — Last Resort Containers

<div> is a block container with zero semantic meaning. <span> is an inline container with zero semantic meaning. Use them only when no semantic tag fits. 

<!-- div: wraps a block of content -->
<div class="card">
  <h3>Title</h3>
  <p>Description</p>
</div>

<!-- span: targets part of inline text -->
<p>
  Price: <span class="price">$29</span>
</p>

<!-- WRONG: block inside inline -->
<span><p>Invalid HTML!</p></span>

Ask first: is there a semantic tag for this? header? nav? section? article? aside? footer? Only use div/span if the answer is no.

header / main / footer / aside / article

The full semantic layout toolkit — every page should use these instead of a sea of divs. 

<header>
  <!-- Logo, site title, main nav -->
</header>

<main>
  <!-- ONE per page. Primary content. -->

  <article>
    <!-- Standalone: blog post, card -->
  </article>

  <aside>
    <!-- Sidebar, related links -->
  </aside>
</main>

<footer>
  <!-- Copyright, bottom nav -->
</footer>

Only one <main> per page. header and footer can appear inside article/section too, not just at the top/bottom of the page.

figure / figcaption / blockquote

For images with captions and quoted content — more descriptive than wrapping everything in a div or p. 

<!-- Image with caption -->
<figure>
  <img src="chart.png" alt="Sales chart">
  <figcaption>
    Fig 1: Sales grew 40% in Q3.
  </figcaption>
</figure>

<!-- Quoted content -->
<blockquote cite="https://source.com">
  <p>"The best way to learn is to build."</p>
  <footer>
    — <cite>Someone Smart</cite>
  </footer>
</blockquote>

figure isn't just for images — it works for code blocks, charts, poems, or any content where a caption makes sense.

title — Tooltips (Beginner-Friendly Extra)

The title attribute adds a tooltip that appears when the user hovers over an element. It works on almost any HTML tag and requires zero JavaScript. This is the one "extra" attribute worth learning now. 

<!-- Hover tooltip on any element -->
<p title="This shows on hover">
  Hover over me!
</p>

<!-- Great for abbreviations -->
<abbr title="HyperText Markup Language">
  HTML
</abbr>

<!-- Works on links too -->
<a href="about.html"
   title="Learn more about our team">
  About Us
</a>

Coming later (don't worry about these now): data-* attributes store custom info for JavaScript to read. aria-label helps screen readers describe elements. You'll learn both naturally when you get to JavaScript and accessibility.

Tips & Tricks — Things Beginners Wish They Knew Earlier

Always connect label to input

Give the input an id, then put the same value in the label's for attribute. Example: <label for="email"> + <input id="email">. Now clicking the label text automatically focuses the input field. Without this, the label is just decoration — it does nothing useful.

Use semantic tags instead of divs

A <nav> tells the browser, screen readers, and search engines "this is a navigation menu." A <div> tells them absolutely nothing. Every time you write a div, ask yourself: is there a more specific tag for this? Usually the answer is yes — <header>, <section>, <article>, <footer>, etc. Semantic HTML is free SEO and free accessibility.

alt text on every image — no exceptions

When an image fails to load, the browser shows the alt text. Screen readers (used by blind users) read it aloud. Search engines index it for image search. If you skip it, your page is broken for millions of users and invisible to Google. If an image is purely decorative (like a background swirl), use alt="" — an empty alt, not a missing one.

Only one h1 per page — then go in order

Your <h1> is the main title of the page — use it once. After that, use <h2> for section titles, <h3> for sub-sections inside those, and so on. Never jump from <h1> straight to <h3> — that's like skipping chapters in a book. Search engines read heading order to understand your page structure.

target="_blank" is a security risk without rel

When you open a link in a new tab with target="_blank", the new page can actually reach back and change your original page using window.opener. This is a real attack vector. The fix is always adding rel="noopener noreferrer" right next to it. This blocks the new tab from having any reference back to your page. Make it a muscle memory habit.

id = one unique thing. class = a group of things

Think of id like a passport number — it belongs to one single person (element) on the page. Think of class like a job title — many people can share it. Use id when you need to jump to something with an anchor link (<a href="#contact">) or target one specific element in CSS/JS. Use class when you want to style or select multiple elements the same way.

Use type, required, and placeholder on every input

type="email" makes the browser validate that the user typed something with an @ sign — for free, no JavaScript needed. required prevents the form from submitting if the field is empty. placeholder="e.g. john@email.com" shows a hint inside the box that disappears when the user starts typing. These three attributes together give you a much better form with zero extra code.

Indent your code — it's not optional

Every child element should be indented one level (2 or 4 spaces) inside its parent. This makes nesting errors obvious at a glance — a missing </div> or </ul> stands out immediately when your code is properly indented. Messy code hides bugs. In VS Code, press Shift+Alt+F to auto-format the whole file in one keystroke.

Validate your HTML before moving on

Go to validator.w3.org and paste your HTML. It will show every error — unclosed tags, wrong nesting, missing attributes. Beginners often have 10+ silent errors they don't know about. The browser tries to fix bad HTML automatically, which means your page might look fine locally but break in a different browser. Always validate before calling a project done.

File names: lowercase, no spaces, use hyphens

Name your files about-me.html, not About Me.html. Spaces in file names break links on servers (Linux is case-sensitive, Windows is not — so Index.html works locally but fails when deployed). Always use lowercase letters, hyphens instead of spaces, and no special characters. This applies to image files, CSS files, and folders too.

Comments explain WHY, not WHAT

HTML comments (<!-- like this -->) are invisible to users but visible to anyone who reads your source code. Use them to mark the start of major sections so you can navigate your file easily: <!-- ===== HERO SECTION ===== -->. Don't use them to explain obvious code — <!-- this is a paragraph --> adds no value. Explain why something is structured the way it is.

Use the browser DevTools constantly

Right-click any element on any webpage → Inspect. You can see the exact HTML structure, hover over elements to see what they are, and even edit HTML live in the browser to experiment. Press F12 to open DevTools. The Elements tab is your best friend for debugging layout problems. Get comfortable in DevTools early — professional developers use it every single day.

Common Mistakes — What Goes Wrong and How to Fix It
✗

Forgetting to close tags. You open <p> but never write </p>. The browser tries to guess where the tag ends, and it often guesses wrong. Everything below the unclosed tag may be swallowed inside it and break your layout completely. This is the #1 beginner bug.

✓ Fix: Every time you open a tag, immediately type the closing tag before writing any content inside. Most code editors like VS Code auto-close tags for you. Self-closing tags like <img>, <br>, <hr>, <input>, and <meta> are the only ones that don't need a closing tag.
✗

Using <div> for absolutely everything. Beginners often write pages with 30+ nested divs and no other tags. This works visually but makes your page invisible to search engines and unusable by screen readers. A div has zero meaning — it's just a box. Developers who see a page full of divs immediately know the author is a beginner.

✓ Fix: Before writing a div, ask: does HTML have a tag that describes this content? Top of page → <header>. Navigation links → <nav>. Main content → <main>. A blog post → <article>. A group of related content → <section>. Bottom of page → <footer>. Only use div when nothing else fits.
✗

Putting block elements inside inline elements. Writing <span><p>text</p></span> or <a><div>...</div></a> is invalid HTML. Inline elements are designed to sit inside text flow. Block elements always start on a new line. Putting a "house" inside a "word" doesn't make sense — the browser will silently fix it in unexpected ways.

✓ Fix: Remember the rule — block elements can contain inline elements, but not the other way around. <p><strong>Bold</strong></p> is fine. <strong><p>...</p></strong> is broken. If you're unsure, run it through the W3C HTML Validator.
✗

Missing the DOCTYPE declaration. Without <!DOCTYPE html> as the very first line, browsers switch into "quirks mode" — a legacy compatibility mode where they render pages the way browsers did in the 1990s. Layouts break, spacing behaves weirdly, and things look different across browsers. You won't see an error message. It just silently breaks.

✓ Fix: Line 1 of every HTML file must always be exactly <!DOCTYPE html>. Nothing before it — not even a blank line or a comment. It tells the browser: "This is modern HTML5, render it correctly."
✗

Using <br> tags to add spacing between sections. Typing <br><br><br> to push content down the page is using the wrong tool. br creates a line break inside text content — like pressing Enter in a paragraph. It's not a spacing tool. Using it for layout is like using tape instead of nails to hang a shelf.

✓ Fix: In HTML, <br> is only correct for actual line breaks inside content — like a postal address ("123 Main St<br>New York") or a poem. When you later learn CSS, you'll discover that spacing between elements is controlled with margin and padding — but that's the next step. For now, just know: br is NOT a spacer.
✗

Wrong or broken file paths for images and links. A file path tells the browser where to find a file — like giving someone directions. If the directions are wrong, the browser can't find the file and shows a broken image or a 404 error. Getting paths wrong is the #1 cause of "my image isn't showing" problems.

📁 my-website/ ├── index.html ← you are here ├── about.html ├── 📁 images/ │ ├── dog.jpg │ └── logo.png └── 📁 pages/ └── contact.html
✓ From index.html, paths work like this:
Same folder → about.html
In a subfolder → images/dog.jpg
In a deeper subfolder → pages/contact.html
From contact.html UP to root → ../index.html or ../images/dog.jpg

Important: Paths are case-sensitive on live servers — Dog.jpg ≠ dog.jpg. Always use lowercase filenames.
✗

Skipping heading levels (h1 → h3, skipping h2). Headings are not just for making text big or small — they create a document outline. Search engines and screen readers read this outline to understand your page. Skipping levels is like writing a book with Chapter 1 followed immediately by Chapter 1.1 with no Section 1 in between. The structure is broken.

✓ Fix: Always go in order: h1 → h2 → h3. You can go back up (h3 → h2 for the next section) but never skip a level going down. Choose a heading level based on its role in the document — not based on how big or small you want it to look. Text size is something CSS controls, which you'll learn next.
✗

Labels that aren't connected to their inputs. Writing <label>Name</label><input type="text"> looks correct in a browser. But the label and the input have no relationship. Clicking the label text does nothing. Screen readers can't associate them. It's visually fine but functionally broken.

✓ Fix: Add matching for and id attributes: <label for="username">Name</label><input id="username" type="text">. Now they're linked. Clicking "Name" focuses the input. Screen readers announce "Name, text field." This is required for accessible, professional forms.
✗

Writing image alt text that's useless. alt="image" or alt="photo" tells a blind user or search engine absolutely nothing. It's technically correct HTML but practically useless. The entire point of alt text is to describe what the image shows so people who can't see it still understand the content.

✓ Fix: Describe what the image actually shows: alt="A golden retriever running on a beach at sunset". Be specific. If the image is a chart, describe what the data shows. If it's a button icon, describe the action: alt="Search". If the image is purely decorative, use alt="" so screen readers skip it.
✗

Writing HTML in UPPERCASE tags. Some beginners write <P>Text</P> or <DIV CLASS="card">. This is technically valid in HTML5 but goes against every modern standard and makes your code look old and inconsistent. Mixing cases makes code much harder to read.

✓ Fix: Always write HTML tags and attributes in lowercase — <p>, <div class="card">, <img src="...">. This is the universal convention in modern HTML and is enforced in XHTML. Lowercase = professional.
Summary — What You Now Know About HTML
01

HTML is Structure, Not Design

HTML defines what content IS — a heading, a paragraph, a link, a list. It doesn't control how it looks. That's CSS's job. Never try to use HTML tags to make things look a certain way.

02

Every Page Needs the Same Skeleton

DOCTYPE → html → head → body. The head holds invisible info (title, charset, links to CSS). The body holds everything the user sees. This structure is never optional.

03

Tags Wrap Content

An opening tag <p> and closing tag </p> wrap the content between them. Some tags are self-closing (img, br, input) and need no closing tag. Nesting = putting tags inside tags.

04

Attributes Give Tags Extra Info

Attributes go inside the opening tag: <a href="...">, <img src="..." alt="...">, <input type="email" required>. They always follow the format name="value".

05

Semantic Tags Have Meaning

header, nav, main, section, article, aside, footer — these tell the browser, search engines, and screen readers what role the content plays. Always prefer these over divs.

06

Lists Are Everywhere

ul for unordered (bullets), ol for ordered (numbers), li for each item. Navigation menus are just styled lists. Feature lists, step-by-step instructions, FAQs — they're all lists under the hood.

07

Links Connect the Web

The a tag with href is what makes HTML a hypertext system. Links can go to other pages, other sites, sections on the same page (#id), email addresses, phone numbers, or downloadable files.

08

Forms Collect User Input

Every input needs a type (what kind of data), a name (label for the server), and a connected label (for accessibility). form wraps it all. button submits it. required and placeholder make it user-friendly.

09

id vs class — Know the Difference

id is a unique identifier for one element — used for anchor links and JS. class is a reusable label for many elements — used for CSS styling. Never duplicate an id on the same page.

10

Accessibility is Built into HTML

alt on images, label connected to inputs, semantic tags, heading order — these aren't "extras." They make your page usable for blind users, keyboard-only users, and search engine bots. They cost nothing to do right.

11

The Browser Fixes Your Mistakes (Silently)

If you write bad HTML, the browser usually renders something anyway. This is dangerous — it makes you think your code is fine when it's not. Always validate at validator.w3.org to find hidden errors.

12

HTML Alone Has No Style — and That's Okay

A page with only HTML looks plain — black text on white background, default fonts. That's completely normal and correct. You are building the foundation. CSS comes next to make it look great.

What's Next After HTML?

You now understand the skeleton of every webpage. Your HTML pages look plain right now — black text, no colour, no layout. That's completely normal and correct. HTML's job is structure, not appearance. Here's the exact path forward:

Step 1 — Done ✓

HTML

You know how to structure content with tags, create links, forms, tables, lists, and use the right semantic elements. This is your foundation — everything else builds on it.

Step 2 — Up Next

CSS

CSS controls how your HTML looks — colours, fonts, spacing, layout, animations. You'll connect a CSS file to your HTML with one line: <link rel="stylesheet" href="style.css"> in your <head>.

Step 3 — After CSS

JavaScript

JavaScript makes your pages interactive — buttons that do things, menus that open, forms that validate, data that loads from the internet. It reads and changes your HTML live in the browser.

Right Now — Do This

Build the 5 HTML Projects Below

Don't skip to CSS yet. Build all 5 HTML-only projects first. Each one forces you to use a different set of tags. Getting comfortable writing HTML without looking things up is the goal.

Useful Tool

W3C HTML Validator

After finishing any project, go to validator.w3.org, paste your HTML, and fix every error it finds. This catches silent bugs the browser hides from you. Run it on every project before moving on.

Useful Tool

Browser DevTools

Press F12 on any webpage to open DevTools. Click the Elements tab to see the live HTML structure. Hover over elements to highlight them. You can even edit HTML live in the browser to experiment — nothing saves, it resets on refresh.

💡 How to Actually Learn From Projects

Start Without Looking

Try to build it from memory first. Only look at the guide when you're genuinely stuck — not to copy code.

Break It Into Pieces

Don't try to build everything at once. Write one function. Make it work. Then the next. Small wins compound.

Make It Ugly First

Get the functionality working first, then style it. Chasing perfect CSS before logic works is the #1 beginner trap.

Add Your Own Feature

Once the base project works, add one feature you came up with yourself. That's when real learning happens.

Put It On GitHub

Push every project to a GitHub repo. After 15 projects you have a real portfolio. Employers look at this.

Rebuild It 30 Days Later

Come back to a finished project a month later and rebuild it from scratch without looking. You'll be amazed how much you remember.
