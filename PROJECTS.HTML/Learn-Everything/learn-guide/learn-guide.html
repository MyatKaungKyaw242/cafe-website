01

grid-template-areas

The Page Skeleton

What is it?

It's like drawing a blueprint before building a house. You name each section of your page and arrange them visually in your CSS.

Real World Example

Think of a newspaper layout — header at top, big story in the middle, smaller stories on the side, footer at the bottom. grid-template-areas lets you draw exactly that.

body {
  display: grid;
  grid-template-areas:
    "header  header"
    "nav     nav"
    "sidebar main"
    "footer  footer";
  grid-template-columns: 250px 1fr;
  grid-template-rows: 70px 50px 1fr 60px;
  min-height: 100vh;
}

header  { grid-area: header; }
nav     { grid-area: nav; }
.sidebar { grid-area: sidebar; }
main    { grid-area: main; }
footer  { grid-area: footer; }

⚠️ Common Beginner Mistakes

  • Forgetting to add grid-area: name on the HTML element
  • Making rows unequal — e.g. 3 words in row 1 but 2 in row 2
  • Not adding display: grid on the parent
  • Using the same name twice for different elements

⚡ Quick Summary

  • Draws your page like a blueprint
  • Every row must have the same number of columns
  • You must assign grid-area to each HTML element
  • Repeated names = element spans those columns
  • A . means leave that cell empty
  • Only works on the parent with display: grid
  • Use it for the big structure only, not small components
  • Think of it as the skeleton before adding muscles

  • Header across the full top
  • Nav bar below header
  • A wide main article on the left
  • A narrow sidebar on the right
  • Footer across the full bottom

Give each area a different background color so you can see the layout clearly. Don't worry about real content yet — just the structure.

02

grid-template-columns

Controlling Column Widths

What is it?

After placing sections on the grid, you control how wide each column is. Think of a spreadsheet — you can drag column widths to make column A narrow and column B wide.

/* Sidebar + main content */
body {
  grid-template-columns: 250px 1fr;
}

/* 4 equal product columns */
.products {
  display: grid;
  grid-template-columns: repeat(4, 1fr);
}

/* Dashboard: sidebar | content | widgets */
.dashboard {
  display: grid;
  grid-template-columns: 200px 1fr 300px;
}

Units Explained

UnitMeaningExample
pxFixed size250px
frFraction of remaining space1fr 2fr
%Percentage of parent50%
autoFits content sizeauto
minmax()Between min and maxminmax(200px, 1fr)
repeat()Repeat a patternrepeat(3, 1fr)

⚠️ Common Beginner Mistakes

  • Using % instead of fr — % ignores the gap
  • Forgetting repeat() and writing 1fr 1fr 1fr 1fr manually
  • Too many fixed px columns — breaks on smaller screens
  • Not understanding fr fills what's left after fixed columns

⚡ Quick Summary

  • Defines how wide each column is
  • fr is the most flexible unit — use it most
  • repeat(n, size) saves repeating the same value
  • minmax(min, max) gives flexible size range
  • px = rigid, fr = flexible
  • auto sizes to the content
  • 1fr 2fr means second column is twice as wide
  • Always test on different screen sizes

03

grid-template-rows

Controlling Row Heights

What is it?

Same as columns but for height of each row. Think of a building floor plan — ground floor is the lobby (tall), upper floors are offices (equal), top floor is the penthouse (tallest).

body {
  display: grid;
  grid-template-rows:
    70px      /* header — short and fixed */
    50px      /* nav bar */
    100vh     /* hero — full screen height */
    auto      /* content — grows with content */
    200px     /* CTA section */
    80px;     /* footer */
}

⚠️ Common Beginner Mistakes

  • Setting all rows to fixed px — content overflows when text is long
  • Forgetting auto — it lets the row grow naturally
  • Using 100vh on multiple rows — only the hero should be full screen
  • Not knowing rows auto-create when you have more content than defined rows

⚡ Quick Summary

  • Defines how tall each row is
  • auto is the safest — grows with content
  • 100vh = full screen height, use only for hero
  • Fixed px rows can overflow — use carefully
  • Rows without a defined height default to auto
  • Works hand-in-hand with grid-template-areas
  • Think of rows like floors in a building
  • Use min-height instead of height to prevent overflow

  • Hero section: 100vh with centered text and a button
  • Features section: auto with 3 feature boxes side by side
  • Footer: 80px

04

grid-auto-rows & grid-auto-columns

Auto Generated Cells

What is it?

When grid creates extra rows automatically (because you have more items than defined), grid-auto-rows controls the size of those auto-created rows. Think of a photo gallery — you don't know how many photos a user will upload.

/* Photo gallery */
.gallery {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  grid-auto-rows: 250px;
  gap: 16px;
}

/* Card grid with flexible height */
.cards {
  display: grid;
  grid-template-columns: repeat(4, 1fr);
  grid-auto-rows: minmax(150px, auto);
  /* at least 150px tall, grows if content is longer */
}

⚠️ Common Beginner Mistakes

  • Confusing grid-template-rows with grid-auto-rows
  • Setting a fixed px without minmax() — content overflows
  • Forgetting it exists — manually defining every row
  • Not using minmax(150px, auto) as the default safe value

⚡ Quick Summary

  • Controls height of automatically created rows
  • Use when you don't know how many rows you'll have
  • minmax(150px, auto) is the safest value
  • Perfect for card grids, galleries, blog posts
  • Works automatically — no need to count rows manually
  • grid-auto-columns does the same for auto columns
  • Think of it as a default row height rule for extra rows
  • Always pair with grid-template-columns

  • Make some cards have more text than others to test the auto height
Goal: See how grid-auto-rows handles different content lengths automatically.
05

Flexbox

Aligning Items Inside Components

What is it?

Flexbox works in one direction — either a row or a column. Use it inside grid cells to align and position smaller elements like buttons, text, icons, nav links. Think of a toolbar in a phone — all icons arranged in a horizontal row, evenly spaced.

/* Navbar */
nav {
  display: flex;
  align-items: center;
  justify-content: space-between;
  gap: 24px;
}

/* Center a button in a section */
.hero {
  display: flex;
  flex-direction: column;
  align-items: center;
  justify-content: center;
}

/* Card with icon + text side by side */
.card {
  display: flex;
  align-items: flex-start;
  gap: 16px;
}

Key Properties

PropertyWhat it does
flex-directionrow (horizontal) or column (vertical)
justify-contentAligns along the main axis
align-itemsAligns along the cross axis
gapSpace between items
flex-wrapAllows items to wrap to next line
flex: 1Item grows to fill available space

⚠️ Common Beginner Mistakes

  • Confusing justify-content and align-items — they swap when you change flex-direction
  • Using flexbox for the whole page layout instead of just components
  • Forgetting flex-wrap: wrap — items overflow instead of wrapping
  • Using margin on every item instead of gap

⚡ Quick Summary

  • Works in one direction — row or column
  • justify-content = align along the main direction
  • align-items = align along the opposite direction
  • Use gap instead of margin for spacing
  • flex: 1 makes an item fill all remaining space
  • flex-wrap: wrap lets items move to the next line
  • Best for navbars, cards, buttons — not full page layouts
  • Think of it as the furniture arranger inside each grid room

  • Profile card: Avatar circle on the left, name + bio text on the right, a follow button at the bottom right of the card

06

gap

Spacing Between Items

What is it?

gap adds space between grid or flex items — like gutters between columns in a magazine. It does NOT add space on the outer edges.

.grid {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  gap: 24px;           /* same gap everywhere */
}

/* Control each direction separately */
.grid {
  row-gap: 32px;       /* only between rows */
  column-gap: 16px;    /* only between columns */
}

⚠️ Common Beginner Mistakes

  • Using margin on every item instead of gap on the parent
  • Forgetting gap doesn't add space on outside edges — use padding on parent
  • Using old grid-gap — use gap instead
  • Setting different gaps per item with margins when column-gap / row-gap would work

⚡ Quick Summary

  • Adds space between items, not outside them
  • Use gap on the parent, not margin on children
  • row-gap and column-gap control each direction
  • Works on both grid and flexbox
  • Replaces the old grid-gap
  • For outer spacing use padding on the container
  • Keeps spacing consistent and easy to update
  • One gap value beats dozens of individual margins

  • Outer padding on the container so the gallery doesn't touch the screen edges
  • At least 12 "photo" boxes
Goal: Feel the difference between gap, row-gap, column-gap, and padding.
07

Media Queries

Responsive Layout

What is it?

Media queries let you change the layout based on screen size. Think of water — it changes shape to fit whatever container it's in. Media queries make your website do the same.

/* Desktop: 4 columns */
.cards {
  display: grid;
  grid-template-columns: repeat(4, 1fr);
}

/* Tablet */
@media (max-width: 1024px) {
  .cards {
    grid-template-columns: repeat(2, 1fr);
  }
}

/* Mobile */
@media (max-width: 768px) {
  .cards {
    grid-template-columns: 1fr;
  }

  body {
    grid-template-areas:
      "header"
      "nav"
      "main"
      "sidebar"
      "footer";
    grid-template-columns: 1fr;
  }
}

/* IMPORTANT: always include this in HTML */
/* meta name="viewport" content="width=device-width, initial-scale=1.0" */

⚠️ Common Beginner Mistakes

  • Only designing for desktop and adding mobile as afterthought
  • Using too many breakpoints — 2 or 3 is usually enough
  • Forgetting the viewport meta tag in HTML
  • Changing too many properties inside media queries

⚡ Quick Summary

  • Changes layout at specific screen widths
  • Most common breakpoints: 768px mobile, 1024px tablet
  • Always include the viewport meta tag in HTML
  • Mobile-first = write mobile styles first, add desktop with min-width
  • Desktop-first = write desktop first, shrink with max-width
  • Keep media query changes small and targeted
  • Test on real phone sizes using Chrome DevTools
  • 2-3 breakpoints covers 99% of real world cases

position

Placing Elements Outside the Normal Flow

What is it?

By default every element sits in the normal flow. position lets you break out of that flow and place elements exactly where you want.

/* relative — nudge from original spot */
.badge {
  position: relative;
  top: -5px;
  left: 10px;
}

/* absolute — placed inside nearest positioned parent */
.card { position: relative; }
.card .sale-tag {
  position: absolute;
  top: 10px;
  right: 10px;
}

/* fixed — always visible, never scrolls away */
.navbar {
  position: fixed;
  top: 0;
  left: 0;
  width: 100%;
  z-index: 1000;
}

/* sticky — scrolls then sticks at threshold */
.section-title {
  position: sticky;
  top: 0;
  background: white;
}

The 5 Values

ValueBehavior
staticDefault — normal flow, ignore it
relativeNudge from original spot, stays in flow
absoluteRemoved from flow, placed inside positioned parent
fixedAlways visible in same spot on screen
stickyFlows normally then sticks at a threshold

⚠️ Common Beginner Mistakes

  • Using absolute without position: relative on parent — element flies to page corner
  • Forgetting width: 100% on a fixed navbar
  • Using fixed when they meant sticky
  • Using position for layouts — that's grid/flexbox's job

⚡ Quick Summary

  • static — default, ignore it
  • relative — nudge + anchor point for absolute children
  • absolute — removed from flow, inside positioned parent
  • fixed — always visible in same screen position
  • sticky — flows then sticks at a scroll threshold
  • Always set position: relative on parent when using absolute on child
  • Position is for exceptions — not for main layout
  • Think of it as placing stickers on top of your normal layout

z-index

Who Appears On Top

What is it?

When elements overlap, z-index decides which one is on top. Think of it as layers in Photoshop — higher number = on top.

.modal-overlay {
  position: fixed;
  z-index: 1000;
}

.navbar {
  position: fixed;
  z-index: 100;
}

.content {
  position: relative;
  z-index: 1;
}

/* Good z-index scale system */
:root {
  --z-base:     1;
  --z-dropdown: 100;
  --z-sticky:   200;
  --z-modal:    300;
  --z-toast:    400;
}

⚠️ Common Beginner Mistakes

  • Using random huge numbers like z-index: 99999
  • Forgetting to add position — z-index silently does nothing on static elements
  • Not understanding stacking context — child can't escape parent's layer
  • Setting z-index on every element — only use where elements overlap

⚡ Quick Summary

  • Higher z-index = appears on top
  • Only works on elements with position that is not static
  • Use a scale system — don't use random numbers
  • Stacking context traps children within parent's layer
  • Use 1, 100, 200, 300 not 1, 99, 999, 9999
  • Modals should always have the highest z-index
  • Navbars need z-index to stay above scrolling content
  • When something disappears behind another — check z-index first

10

transitions & animations

Making it Feel Alive

What is it?

transition smoothly changes a CSS property over time. animation creates complex multi-step movements using @keyframes.

/* Transition — smooth hover effect */
.button {
  background: #e94560;
  transform: translateY(0);
  box-shadow: 0 2px 8px rgba(0,0,0,0.1);
  transition:
    background 0.3s ease,
    transform 0.3s ease,
    box-shadow 0.3s ease;
}

.button:hover {
  background: #c73652;
  transform: translateY(-3px);
  box-shadow: 0 8px 20px rgba(0,0,0,0.2);
}

/* Animation — multi-step movement */
@keyframes fadeInUp {
  from {
    opacity: 0;
    transform: translateY(30px);
  }
  to {
    opacity: 1;
    transform: translateY(0);
  }
}

.hero-text {
  animation: fadeInUp 0.6s ease-out forwards;
}

/* Infinite pulse */
@keyframes pulse {
  0%, 100% { transform: scale(1); }
  50%       { transform: scale(1.06); }
}

.notification {
  animation: pulse 1.5s ease-in-out infinite;
}

⚠️ Common Beginner Mistakes

  • Transitioning width or height — use transform instead
  • Using transition: all — transitions every property, causes bugs
  • Forgetting forwards on animations — element snaps back
  • Making animations too long — over 0.5s feels slow

⚡ Quick Summary

  • transition = smooth change between two states
  • animation = multi-step movement with @keyframes
  • Always specify which property to transition — not all
  • ease is the most natural timing
  • Keep transitions short — 0.2s to 0.4s is the sweet spot
  • Use transform for movement — never animate width or height
  • forwards keeps the element in its final animation state
  • Subtle animations = professional. Too many = amateur

11

CSS Variables

Your Design System

What is it?

CSS variables let you store values once and reuse them everywhere. Change one variable and it updates across your whole site — perfect for colors, fonts, spacing, and dark mode.

/* Define in :root (globally available) */
:root {
  --primary:    #e94560;
  --dark:       #1a1a2e;
  --light:      #f0f2f5;
  --text:       #333;
  --font-main:  'Segoe UI', Arial, sans-serif;
  --radius:     8px;
  --shadow:     0 4px 12px rgba(0,0,0,0.1);
  --transition: 0.3s ease;
}

/* Use anywhere with var() */
.button {
  background: var(--primary);
  border-radius: var(--radius);
  box-shadow: var(--shadow);
  transition: var(--transition);
}

/* Dark mode with variables */
:root {
  --bg:   #ffffff;
  --text: #333333;
}

body.dark-mode {
  --bg:   #1a1a2e;
  --text: #e0e0f0;
}

body {
  background: var(--bg);
  color: var(--text);
  /* Automatically switches with the class toggle */
}

⚠️ Common Beginner Mistakes

  • Defining variables inside a specific selector — they won't be global
  • Using hardcoded colors alongside variables — defeats the purpose
  • Not using variables for repeated values like spacing and border-radius
  • Naming variables poorly — --color1 tells nothing, --primary does

⚡ Quick Summary

  • Define variables in :root to make them globally available
  • Use var(--name) to apply them anywhere
  • Change one variable = updates everywhere it's used
  • Perfect for colors, fonts, spacing, shadows, border-radius
  • Makes dark mode trivially easy to implement
  • Name them by purpose — --primary not --red
  • Variables make your CSS a proper design system
  • Saves you from find-and-replace nightmares when redesigning

overflow

Controlling What Spills Out

What is it?

When content is bigger than its container, overflow decides what happens — show it, hide it, or make it scrollable.

/* Image zoom that stays inside card */
.card {
  overflow: hidden;
  border-radius: 12px;
}

.card img {
  transition: transform 0.4s ease;
}

.card:hover img {
  transform: scale(1.08);
}

/* Scrollable sidebar */
.sidebar {
  height: 100vh;
  overflow-y: auto;
}

/* Fix horizontal scroll on mobile */
body {
  overflow-x: hidden;
}

The 4 Values

ValueBehavior
visibleDefault — content spills out
hiddenClips content at container edge
scrollAlways shows scrollbar
autoScrollbar only when needed

⚠️ Common Beginner Mistakes

  • Using overflow: hidden to hide a broken layout — fixes symptom not cause
  • overflow: hidden also clips box shadows — shadow disappears at edge
  • Not using overflow-y: auto on fixed-height containers
  • Using overflow: scroll instead of auto — always shows scrollbar

⚡ Quick Summary

  • visible = default, content spills out
  • hidden = clips content at container edge
  • auto = scrollbar only when needed — use this most
  • scroll = always shows scrollbar — rarely what you want
  • hidden clips children's box shadows — be careful
  • Use overflow-y: auto for scrollable panels
  • overflow-x: hidden on body fixes horizontal scroll on mobile
  • Don't use hidden to cover up layout bugs

JavaScript DOM

Adding Interactivity

What is it?

JavaScript lets you change the page after it loads — toggle menus, show modals, switch dark mode, validate forms. Without it your site is just a static picture.

/* Select elements */
const btn = document.querySelector('.btn');
const cards = document.querySelectorAll('.card');

/* Toggle class on click — most used pattern */
btn.addEventListener('click', () => {
  nav.classList.toggle('hidden');
});

/* Dark mode toggle */
const toggle = document.querySelector('.dark-toggle');
toggle.addEventListener('click', () => {
  document.body.classList.toggle('dark-mode');
});

/* Sticky navbar on scroll */
const navbar = document.querySelector('nav');
window.addEventListener('scroll', () => {
  if (window.scrollY > 50) {
    navbar.classList.add('scrolled');
  } else {
    navbar.classList.remove('scrolled');
  }
});

/* Loop through multiple elements */
cards.forEach(card => {
  card.addEventListener('click', () => {
    card.classList.toggle('active');
  });
});

⚠️ Common Beginner Mistakes

  • Using getElementById — querySelector is cleaner
  • Writing onclick directly in HTML — use addEventListener
  • Manipulating CSS directly in JS — toggle classes instead
  • Not adding defer on the script tag — JS runs before HTML loads

⚡ Quick Summary

  • querySelector selects one, querySelectorAll selects many
  • addEventListener('click', fn) responds to interactions
  • classList.toggle() is the most powerful pattern
  • Keep styling in CSS, keep logic in JS — never mix
  • defer on script tag ensures HTML loads before JS runs
  • Dark mode = toggle a class on body + CSS variables do the rest
  • Scroll events trigger effects based on scroll position
  • JS + CSS classes together = almost any interactive effect

14

clamp() / min() / max()

Fluid Sizing Without Media Queries

What is it?

These functions let values scale fluidly between a minimum and maximum — no breakpoints needed. Text and spacing that automatically adapt to any screen size.

/* clamp(min, preferred, max) */
h1 {
  font-size: clamp(1.5rem, 4vw, 3rem);
  /* small screen → 1.5rem */
  /* medium screen → 4vw (scales) */
  /* large screen → 3rem */
}

section {
  padding: clamp(20px, 5vw, 80px);
}

/* min(a, b) — picks the smaller value */
.container {
  width: min(90%, 1200px);
  /* 90% on small, caps at 1200px on large */
  margin: 0 auto;
}

/* max(a, b) — picks the larger value */
.sidebar {
  width: max(200px, 25%);
  /* always at least 200px, grows to 25% on large screens */
}

/* Combined with CSS variables */
:root {
  --font-h1: clamp(1.8rem, 5vw, 4rem);
  --font-h2: clamp(1.4rem, 3vw, 2.5rem);
  --padding: clamp(16px, 4vw, 60px);
}

⚠️ Common Beginner Mistakes

  • Using vw for font sizes without clamp — unreadably tiny or huge
  • Forgetting the order in clamp — always min, preferred, max
  • Using these instead of media queries for layout changes — they're for sizing, not layout
  • Not combining with CSS variables — misses the full power

⚡ Quick Summary

  • clamp(min, preferred, max) — fluid value between a floor and ceiling
  • min(a, b) — always picks the smaller of two values
  • max(a, b) — always picks the larger of two values
  • vw in the preferred value makes it scale with screen width
  • Perfect for font sizes, padding, container widths
  • Reduces the number of media queries you need
  • width: min(90%, 1200px) is the cleanest container pattern
  • Combine with CSS variables for a complete fluid design system

What is it?

CSS Pseudo-classes

Targeting Specific States & Positions

What is it?

Pseudo-classes let you style elements based on their state or position — when hovered, when focused, when it's the first child, when a checkbox is checked. They start with a colon :.

Real World Example

Think of a traffic light — the same light changes appearance based on its state (red, yellow, green). Pseudo-classes do the same — same element, different style based on what's happening to it.

/* State-based pseudo-classes */
a:hover   { color: #e94560; }         /* mouse over */
a:visited { color: purple; }          /* already clicked */
input:focus { border-color: blue; }   /* currently typing */
button:active { transform: scale(0.97); } /* being clicked */

/* Position-based pseudo-classes */
li:first-child { font-weight: bold; }  /* first item */
li:last-child  { border: none; }       /* last item */
li:nth-child(2) { color: red; }        /* 2nd item */
li:nth-child(odd)  { background: #f5f5f5; } /* 1, 3, 5... */
li:nth-child(even) { background: white; }   /* 2, 4, 6... */

/* Useful extras */
p:not(.special) { color: gray; } /* everything except .special */
input:required  { border-left: 3px solid red; }
input:checked + label { color: green; }

⚠️ Common Beginner Mistakes

  • Confusing :nth-child() counts — it counts ALL siblings, not just matching ones
  • Using :first-child when they need :first-of-type
  • Forgetting pseudo-classes work on the element itself, not its parent
  • Not using :focus on inputs — critical for accessibility

⚡ Quick Summary

  • Pseudo-classes start with : and target states or positions
  • :hover = mouse over, :focus = clicked/tabbed into
  • :active = element being clicked right now
  • :nth-child(n) targets by position in parent
  • odd and even inside :nth-child() for alternating styles
  • :not() styles everything except what you specify
  • :first-child and :last-child for edge items
  • Always style :focus on interactive elements for accessibility

Accessibility (a11y)

Building for Everyone

What is it?

Accessibility means building websites that everyone can use — including people who use screen readers, can't use a mouse, or have visual impairments. It's also required by law in many countries for professional websites.

Real World Example

Think of a building with a wheelchair ramp — the ramp doesn't hurt anyone who can walk, but it makes the building usable for people in wheelchairs. Accessibility features are the same — they don't hurt normal users but open your site to millions more people.

<!-- Always add alt text to images -->
<img src="coffee.jpg" alt="A hot latte with leaf art in foam" />

<!-- Decorative image — empty alt so screen reader skips it -->
<img src="divider.png" alt="" />

<!-- ARIA labels for icon-only buttons -->
<button aria-label="Close menu">✕</button>

<!-- Skip to main content link (for keyboard users) -->
<a href="#main" class="skip-link">Skip to main content</a>

/* Never remove focus outline — keyboard users need it */
/* Bad */
button:focus { outline: none; }

/* Good — custom styled focus */
button:focus {
  outline: 2px solid #e94560;
  outline-offset: 3px;
}

/* Sufficient color contrast */
/* Bad — light gray on white */
p { color: #aaa; background: #fff; }

/* Good — dark text on light background */
p { color: #333; background: #fff; }

⚠️ Common Beginner Mistakes

  • Removing outline: none on focus — keyboard users can't see where they are
  • Missing alt on images — screen readers say "image" with no context
  • Low color contrast — light gray text on white background fails accessibility checks
  • Using <div> as a button — not keyboard navigable by default

⚡ Quick Summary

  • Always add alt text to images — describe what's in the image
  • Never remove :focus outline — restyle it but keep it visible
  • Use real <button> and <a> tags — not clickable divs
  • Color contrast ratio must be at least 4.5:1 for normal text
  • aria-label describes elements that have no visible text
  • Use semantic HTML — screen readers depend on it
  • Test keyboard navigation — tab through your site without a mouse
  • Accessibility = better UX for everyone, not just disabled users

SEO Basics

Getting Found on Google

What is it?

SEO (Search Engine Optimization) is how you make Google find, understand, and rank your website. The better your SEO, the higher you appear in search results.

Real World Example

Think of Google as a librarian — when someone searches "best coffee shop", the librarian looks for books (websites) that are clearly labeled, well-organized, and actually about coffee shops. SEO is how you label and organize your site for the librarian.

<head>

  <title>BrewBolt — Premium Coffee Delivery</title>

  <!-- Description — shows under title in Google results -->
  <meta name="description"
    content="Order fresh specialty coffee delivered to your door.
    Free shipping on orders over $30." />

  <!-- Tell Google your page is mobile-friendly -->
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />

  <!-- Canonical URL — prevents duplicate content issues -->
  <link rel="canonical" href="https://brewbolt.com/" />
</head>

<body>
  <!-- One h1 per page — your main keyword -->
  <h1>Fresh Coffee Delivered to Your Door</h1>

  <!-- Descriptive image alt text — Google reads this too -->
  <img src="coffee.jpg" alt="Freshly brewed pour-over coffee" />

  <!-- Descriptive link text — not "click here" -->
  <a href="/menu">Browse Our Coffee Menu</a>
</body>

⚠️ Common Beginner Mistakes

  • Missing or duplicate <title> tags — every page needs a unique title
  • No meta description — Google shows a random snippet instead
  • Using multiple <h1> tags — confuses Google about the main topic
  • Using "click here" as link text — Google can't tell what the link is about

⚡ Quick Summary

  • Every page needs a unique, descriptive <title>
  • Meta description is what shows under your title in Google — keep it under 160 chars
  • One <h1> per page — put your main keyword in it
  • Descriptive image alt text helps Google understand images
  • Use meaningful link text — not "click here" or "read more"
  • Fast loading = better ranking — optimize images
  • Semantic HTML helps Google understand your page structure
  • Mobile-friendly pages rank higher — always use viewport meta tag

Performance

Making Your Site Load Fast

What is it?

Performance is how fast your website loads. Studies show that 53% of users leave a site that takes more than 3 seconds to load. Fast sites also rank higher on Google.

Real World Example

Think of a fast food restaurant vs a slow one — same food, but people choose the faster one. Your website is the same — a slow site drives users to your competitor's faster site.

<!-- 1. Lazy load images — only loads when user scrolls to them -->
<img src="product.jpg" alt="Product" loading="lazy" />

<!-- 2. Use modern image formats -->
<img src="hero.webp" alt="Hero image" />
<!-- WebP = same quality as JPEG but 30% smaller file size -->

<!-- 3. Load CSS in head, JS at bottom with defer -->
<head>
  <link rel="stylesheet" href="style.css" />
</head>
<body>
  ...content...
  <script src="script.js" defer></script>
</body>

/* 4. Use transform instead of top/left for animations */
/* Bad — causes layout recalculation */
.box { left: 100px; }

/* Good — GPU-accelerated, much faster */
.box { transform: translateX(100px); }

/* 5. Use system fonts for fastest load */
body {
  font-family: -apple-system, BlinkMacSystemFont,
               'Segoe UI', Arial, sans-serif;
}

⚠️ Common Beginner Mistakes

  • Using giant unoptimized images — a 5MB hero image kills load time
  • Loading JavaScript in <head> without defer — blocks page render
  • Animating top, left, width — use transform instead
  • Loading huge Google Font libraries with every font weight

⚡ Quick Summary

  • Add loading="lazy" to all images below the fold
  • Use WebP format — same quality, 30% smaller than JPEG
  • Always add defer to script tags
  • Use transform for animations — not top/left/width
  • Compress images before uploading — use squoosh.app
  • Fewer HTTP requests = faster site — combine files when possible
  • Test performance with Google Lighthouse in Chrome DevTools
  • A score above 90 in Lighthouse = fast, well-optimized site

JavaScript Beyond DOM

Fetch, Local Storage & Form Validation

What is it?

Beyond toggling classes, JavaScript can fetch data from the internet, remember user preferences, and validate forms before submission — skills every real website uses.

Real World Example

Think of a weather app — it fetches today's weather from a server (Fetch API), remembers your city preference next time you open it (localStorage), and shows an error if you type letters instead of a zip code (form validation).

// 1. FETCH API — get data from the internet
fetch('https://api.example.com/products')
  .then(response => response.json())
  .then(data => {
    data.forEach(product => {
      console.log(product.name);
    });
  });

// 2. LOCAL STORAGE — save data in the browser
// Save
localStorage.setItem('theme', 'dark');
localStorage.setItem('username', 'Myat');

// Read
const theme = localStorage.getItem('theme');
document.body.classList.add(theme);

// Remove
localStorage.removeItem('theme');

// 3. FORM VALIDATION — check before submitting
const form = document.querySelector('form');
const email = document.querySelector('#email');

form.addEventListener('submit', (e) => {
  e.preventDefault(); // stop form from submitting

  if (email.value === '') {
    showError('Email is required');
    return;
  }

  if (!email.value.includes('@')) {
    showError('Please enter a valid email');
    return;
  }

  // All good — submit the form
  console.log('Form submitted!');
});

function showError(message) {
  const error = document.querySelector('.error');
  error.textContent = message;
  error.style.display = 'block';
}

⚠️ Common Beginner Mistakes

  • Forgetting e.preventDefault() — page refreshes on form submit before you can validate
  • Not handling fetch errors — app breaks silently when API fails
  • Storing sensitive data in localStorage — it's readable by anyone, never store passwords
  • Only relying on frontend validation — always validate on the server too

⚡ Quick Summary

  • Fetch API gets data from external servers — returns a Promise
  • Always .then(res => res.json()) to convert response to usable data
  • localStorage saves key-value pairs in the browser permanently
  • localStorage survives page refresh — sessionStorage clears on tab close
  • Always call e.preventDefault() first in form submit handler
  • Never store passwords or tokens in localStorage — use cookies or sessions
  • Show clear error messages next to the specific input that failed
  • Frontend validation = good UX. Backend validation = real security

Git & Version Control

Never Lose Your Work Again

What is it?

Git is a tool that tracks every change you make to your code. Think of it as a save system with infinite undo — you can go back to any point in history. GitHub is a website where you store your Git projects online so they're backed up and shareable.

Real World Example

Think of Google Docs version history — every time you make changes, a snapshot is saved. You can see who changed what and go back to any previous version. Git does exactly this for your code, but you control when snapshots are saved (called commits).

The 5 Commands You Use Every Day

# 1. Start tracking a project
git init

# 2. Check what files have changed
git status

# 3. Stage files — tell Git "include these in the next save"
git add .                    # stage all changed files
git add index.html           # stage one specific file

# 4. Commit — save a snapshot with a message
git commit -m "Add hero section to homepage"

# 5. Push — upload your commits to GitHub
git push origin main

The Full Beginner Workflow

# First time — connect local project to GitHub
git init
git add .
git commit -m "First commit"
git branch -M main
git remote add origin https://github.com/yourname/project.git
git push -u origin main

# Every day after that — just these 3 commands
git add .
git commit -m "Describe what you changed"
git push

# Undo last commit (keeps your files)
git reset --soft HEAD~1

# See full history of commits
git log --oneline

# Go back to a previous version
git checkout abc1234    # use the commit ID from git log

Branches — Work Without Breaking Things

# Create a new branch for a new feature
git checkout -b new-navbar

# Work on your feature, commit changes...
git add .
git commit -m "Redesign navbar with mobile menu"

# Switch back to main branch
git checkout main

# Merge your feature into main when it's ready
git merge new-navbar

# Delete the feature branch — no longer needed
git branch -d new-navbar

⚠️ Common Beginner Mistakes

  • Not committing often enough — losing hours of work if something goes wrong
  • Vague commit messages like "fix stuff" — useless when looking back at history
  • Working directly on the main branch — always create a branch for new features
  • Not having a .gitignore file — accidentally uploading node_modules or sensitive files

⚡ Quick Summary

  • Git tracks every change — infinite undo for your code
  • The 3 daily commands: git add . → git commit -m → git push
  • Commit often — after every small working change
  • Write clear commit messages — "Add mobile navbar" not "fix"
  • GitHub = online backup + portfolio for your projects
  • Branches let you work on features without breaking the main code
  • git status — always run this to see what's changed
  • git log --oneline — see your full project history at a glance

  • In your project folder, run git init

Goal: Get your code on GitHub and make your first 2 commits. This is the workflow you'll use every day as a developer.

Hosting & Deployment

Putting Your Site on the Internet

What is it?

Right now your website only lives on your computer. Hosting means uploading it to a server so anyone in the world can visit it with a URL. Deployment is the process of doing that upload — and the best platforms make it automatic.

Real World Example

Think of your website as a shop you built in your garage. It's complete and beautiful — but no one can visit because it's in your garage. Hosting is renting a space on a real street (a server) so customers can find you. Deployment is moving everything from your garage to that street location.

Best Free Hosting Options for Beginners

PlatformBest forHow it works
GitHub PagesStatic HTML/CSS/JS sitesPush to GitHub → site goes live automatically
NetlifyAny frontend projectDrag and drop your folder — site is live in 30 seconds
VercelFrontend + JavaScript frameworksConnect GitHub repo → auto-deploys on every push

Method 1 — GitHub Pages (Easiest)

# Step 1: Push your project to GitHub (see Topic 25)

# Step 2: In your GitHub repo:
# Settings → Pages → Source → "Deploy from branch"
# Branch: main → Folder: / (root) → Save

# Step 3: Your site is live at:
# https://yourusername.github.io/your-repo-name/

# Every time you push new code → site updates automatically

Method 2 — Netlify Drop (30 Seconds)

# Step 1: Go to netlify.com/drop
# Step 2: Drag your project folder into the browser window
# Step 3: Done — your site is live at a random URL like:
#         https://random-name-123.netlify.app

# To get a custom URL:
# Create a Netlify account → rename your site
# https://myproject.netlify.app

# Auto-deploy from GitHub:
# Netlify → New Site → Import from Git → pick your repo
# Every git push → Netlify auto-deploys in 30 seconds

Custom Domain — Using Your Own URL

# Free subdomain (no cost):
# yourname.netlify.app
# yourname.github.io

# Netlify → Domain Settings → Add custom domain
# Add DNS records at your domain registrar
# Wait 24 hours for it to go live

⚠️ Common Beginner Mistakes

  • File paths using backslashes \ on Windows — servers use forward slashes /
  • Linking CSS with absolute paths like C:/Users/... — only works on your computer
  • Uploading with wrong file — make sure your HTML file is named index.html for the home page
  • Forgetting to push latest changes before sharing the URL — visitors see an old version

⚡ Quick Summary

  • Hosting = your site lives on a server anyone can reach
  • GitHub Pages = free, automatic, perfect for beginners
  • Netlify = drag and drop your folder — live in 30 seconds
  • Always name your main file index.html — servers look for this by default
  • Use relative paths in HTML — never absolute paths like C:/Users/...
  • Auto-deploy = connect GitHub to Netlify — every push updates your live site
  • Free subdomain: yourname.netlify.app — looks professional enough to share
  • Custom domain costs ~$10/year — worth it for a portfolio
