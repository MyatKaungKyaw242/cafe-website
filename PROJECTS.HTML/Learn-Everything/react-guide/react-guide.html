01

What is React?

Why the whole frontend world uses it

The Problem React Solves

With vanilla JS, as your UI gets complex, you end up manually updating the DOM for every state change. Change a user's name → update the header, the sidebar, the profile card, the greeting. It becomes impossible to manage. React solves this by letting you describe what the UI should look like for any given state — and it handles the DOM updates automatically.

The Core Idea

React is built around one simple idea: UI = f(state). Your UI is a function of your state. Change the state, and React automatically re-renders the right parts of the page. You never manually touch the DOM — you just update state and trust React to keep the UI in sync.

Real World Example

Think of a Google Docs document. Every time you type, the word count updates, the save indicator changes, collaboration cursors move, the auto-complete suggests words — all simultaneously. With vanilla JS this would be hundreds of manual DOM calls. With React, you just update the document state and every component that depends on it updates automatically.

// Vanilla JS — manual DOM for every change
document.getElementById("name").textContent = newName;
document.querySelector(".header-name").textContent = newName;
document.querySelector(".profile-name").textContent = newName;
document.querySelector(".greeting").textContent = "Hi " + newName;

// React — update state, everything syncs automatically
setName(newName); // React handles all the DOM updates

React's Key Concepts (Preview)

ConceptWhat it means
ComponentA reusable piece of UI — like a LEGO block
StateData that can change and drives the UI
PropsData passed from a parent to a child component
HookA special function that adds features to components (useState, useEffect)
JSXHTML-like syntax you write inside JavaScript
Re-renderReact automatically updates the DOM when state changes

Quick Summary

  • React = UI library, not a full framework
  • UI = f(state) — change state, UI updates
  • You describe WHAT, React handles HOW
  • Component-based — build with reusable pieces
  • Most in-demand frontend skill on the job market
  • Built on top of everything you learned in JS

Setup with Vite

Creating a real React project in 2 minutes

What is Vite?

Vite is a build tool — it sets up your development environment so you can write React with modern features (modules, hot reload, bundling). Think of it as the engine that runs under your React project. It replaces the older Create React App which is now outdated.

# Step 1: Open terminal in VS Code (Ctrl + `)
# Step 2: Create a new React project
npm create vite@latest my-app -- --template react

# Step 3: Move into the project
cd my-app

# Step 4: Install dependencies
npm install

# Step 5: Start the dev server
npm run dev
# Opens at http://localhost:5173

Project Structure

my-app/
├── index.html          ← the one HTML file
├── package.json        ← project config + dependencies
├── vite.config.js      ← Vite configuration
├── node_modules/       ← installed packages (don't touch)
└── src/
    ├── main.jsx        ← entry point — mounts React to the DOM
    ├── App.jsx         ← root component
    ├── App.css
    └── assets/

The Entry Point

// src/main.jsx — this is where React starts
import React from 'react'
import ReactDOM from 'react-dom/client'
import App from './App.jsx'
import './index.css'

ReactDOM.createRoot(document.getElementById('root')).render(
  <React.StrictMode>
    <App />
  </React.StrictMode>
)
// This mounts your entire app inside <div id="root"> in index.html

Essential npm Commands

CommandWhat it does
npm run devStart development server (hot reload)
npm run buildBuild for production
npm install package-nameInstall a new package
npm installInstall all dependencies (after cloning)

❌ Common Mistakes

  • Running npm run dev before npm install — node_modules doesn't exist yet
  • Editing files in node_modules/ — they get overwritten on reinstall
  • Committing node_modules/ to Git — add it to .gitignore
  • Using create-react-app — it's outdated and slow. Always use Vite

Quick Summary

  • Vite = fast modern build tool for React
  • npm create vite@latest scaffolds the project
  • npm install then npm run dev
  • All your code lives in src/
  • main.jsx is the entry, App.jsx is the root
  • Hot reload — changes appear instantly without refresh
03

JSX

HTML-like syntax inside JavaScript — React's secret weapon

What is JSX?

JSX lets you write what looks like HTML directly inside your JavaScript. It's not actually HTML — it gets compiled to regular JavaScript by Vite behind the scenes. It's the cleanest way to describe what your UI should look like.

// JSX — what you write
const element = <h1 className="title">Hello, Myat!</h1>;

// What it compiles to (you never write this)
const element = React.createElement("h1", { className: "title" }, "Hello, Myat!");

JSX Rules — These Will Trip You Up

// 1. Return ONE root element (wrap in a div or Fragment)
return (
  <div>
    <h1>Title</h1>
    <p>Paragraph</p>
  </div>
);

// Use Fragment <></> to avoid extra divs
return (
  <>
    <h1>Title</h1>
    <p>Paragraph</p>
  </>
);

// 2. class → className (class is a reserved JS keyword)
<div className="card">...</div>

// 3. Self-close tags that have no children
<img src="photo.jpg" alt="photo" />
<input type="text" />
<br />

// 4. JavaScript expressions in {} — not statements
<p>{name}</p>
<p>{2 + 2}</p>
<p>{isLoggedIn ? "Welcome!" : "Please log in"}</p>
<p>{user.name.toUpperCase()}</p>

// 5. Inline styles use objects with camelCase keys
<div style={{ backgroundColor: "red", fontSize: "16px" }}></div>

// 6. Comments inside JSX
<div>
  {/* This is a JSX comment */}
  <p>Hello</p>
</div>

❌ Common Mistakes

  • Using class instead of className — works but gives a warning
  • Forgetting to close self-closing tags: <img> is wrong, must be <img />
  • Trying to use if/else directly in JSX — use ternary or && instead
  • Returning multiple root elements without a wrapper
  • Using for instead of htmlFor on label elements

Quick Summary

  • JSX = HTML-like syntax in JavaScript
  • Use className not class
  • Must return one root element
  • Use {} for JS expressions
  • Self-close all tags: <img />
  • Use Fragment <></> to avoid extra divs

Components

Reusable building blocks — the heart of React

What is a Component?

A component is a JavaScript function that returns JSX. It's a reusable piece of UI — like a LEGO block. You build your entire app by composing components together. A page is a component. A button is a component. A nav bar is a component made of smaller components.

// A simple component — just a function that returns JSX
const Button = () => {
  return <button className="btn">Click Me</button>;
};

// Component names MUST start with a capital letter
// lowercase = HTML tag, Uppercase = React component
<div>    // HTML element
<Button> // React component

// Using a component inside another component
const App = () => {
  return (
    <div className="app">
      <Button />
      <Button />
      <Button />
    </div>
  );
};

Component File Structure

One component per file. File name matches the component name. Put components in a components/ folder inside src/.

src/
├── components/
│   ├── Button.jsx
│   ├── Card.jsx
│   ├── Navbar.jsx
│   └── Footer.jsx
├── pages/
│   ├── Home.jsx
│   └── About.jsx
├── App.jsx
└── main.jsx

// Button.jsx
const Button = () => {
  return <button className="btn">Click Me</button>;
};
export default Button;

// App.jsx
import Button from "./components/Button";

const App = () => {
  return (
    <div>
      <Button />
    </div>
  );
};
export default App;

❌ Common Mistakes

  • Lowercase component name — React treats it as an HTML tag and it doesn't work
  • Mutating data inside a component directly — always use setState
  • Making components too big — one component, one job. If it's over 100 lines, split it
  • Forgetting to export the component — import fails silently or throws an error

Props

Passing data into components — making them dynamic and reusable

What are Props?

Props (short for properties) are how you pass data from a parent component to a child component. They make components reusable — instead of a Button that always says "Click Me", you pass in the label as a prop and the same Button component can say anything. Props are read-only — a child never changes its own props.

// Defining a component that accepts props
const Button = (props) => {
  return <button className={`btn btn-${props.variant}`}>{props.label}</button>;
};

// Cleaner with destructuring (always do this)
const Button = ({ label, variant = "primary", onClick }) => {
  return (
    <button className={`btn btn-${variant}`} onClick={onClick}>
      {label}
    </button>
  );
};

// Passing props (looks like HTML attributes)
<Button label="Submit"  variant="primary"   onClick={handleSubmit} />
<Button label="Cancel"  variant="secondary" onClick={handleCancel} />
<Button label="Delete"  variant="danger"    onClick={handleDelete} />

Children Prop — Wrapping Content

The special children prop lets you pass JSX content between component opening and closing tags — like how HTML elements can have content inside them.

// Card accepts children
const Card = ({ title, children }) => {
  return (
    <div className="card">
      <h3>{title}</h3>
      <div className="card-body">{children}</div>
    </div>
  );
};

// Usage — anything between the tags becomes children
<Card title="User Profile">
  <img src={avatar} alt="avatar" />
  <p>{user.bio}</p>
  <Button label="Follow" />
</Card>

❌ Common Mistakes

  • Trying to modify props inside a child — props are read-only. Use state instead
  • Passing a function result instead of a function reference: onClick={handleClick()} calls it immediately. Use onClick={handleClick}
  • Forgetting default props — if a prop might be undefined, add a default in destructuring
  • Prop drilling too deep — passing props through 5+ levels. Use Context API instead

Quick Summary

  • Props = data from parent to child
  • Always destructure: ({ name, age })
  • Props are read-only — never modify them
  • children = content between tags
  • Default values in destructuring
  • Pass functions as props for callbacks

useState

Storing data that changes — the most important hook

What is State?

State is data that belongs to a component and can change over time. When state changes, React automatically re-renders the component with the new data. State is the engine of interactivity in React. Examples: a counter, whether a modal is open, the value in an input field, the list of todos.

import { useState } from 'react';

// useState(initialValue) returns [currentValue, setterFunction]
const Counter = () => {
  const [count, setCount] = useState(0); // start at 0

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>+</button>
      <button onClick={() => setCount(count - 1)}>-</button>
      <button onClick={() => setCount(0)}>Reset</button>
    </div>
  );
};

State with Objects and Arrays

When state is an object or array, you must create a new copy — never mutate directly. React compares the old and new state by reference, so if you mutate in place, it won't detect the change and won't re-render.

// State with an object
const [user, setUser] = useState({ name: "Myat", age: 19 });

// WRONG — mutates directly
user.age = 20; // React won't re-render!

// CORRECT — create new object with spread
setUser({ ...user, age: 20 });

// State with an array (todos example)
const [todos, setTodos] = useState(["Learn React", "Build a project"]);

// Add item — spread + new item
setTodos([...todos, "Deploy to Vercel"]);

// Remove item — filter (returns new array)
setTodos(todos.filter(t => t !== "Learn React"));

// Update item — map (returns new array)
setTodos(todos.map(t => t === "Learn React" ? "React mastered!" : t));

Multiple State Variables

const LoginForm = () => {
  const [email,    setEmail]    = useState("");
  const [password, setPassword] = useState("");
  const [isLoading, setIsLoading] = useState(false);
  const [error,    setError]    = useState(null);

  const handleSubmit = async (e) => {
    e.preventDefault();
    setIsLoading(true);
    setError(null);
    try {
      await login(email, password);
    } catch (err) {
      setError(err.message);
    } finally {
      setIsLoading(false);
    }
  };

  return (
    <form onSubmit={handleSubmit}>
      <input value={email}    onChange={e => setEmail(e.target.value)}    />
      <input value={password} onChange={e => setPassword(e.target.value)} type="password" />
      {error && <p className="error">{error}</p>}
      <button disabled={isLoading}>{isLoading ? "Logging in..." : "Login"}</button>
    </form>
  );
};

❌ Common Mistakes

  • Mutating state directly — todos.push(item) won't trigger a re-render. Always use the setter
  • Using state for data that doesn't need to cause a re-render — use useRef instead
  • Setting state inside render (no condition) — creates an infinite re-render loop
  • Reading state immediately after setting it — setState is async. The value won't be updated on the next line

Quick Summary

  • useState(init) returns [value, setter]
  • Calling the setter triggers a re-render
  • Never mutate state directly
  • Use spread to update objects/arrays
  • Each component has its own independent state
  • State is the engine of all interactivity

useEffect

Running code when things change — fetch, timers, subscriptions

What is useEffect?

useEffect lets you run side effects — code that reaches outside of React, like fetching data, setting up a timer, or syncing with localStorage. It runs after the component renders. The dependency array controls when it runs.

import { useState, useEffect } from 'react';

// Runs after EVERY render (no dependency array) — usually not what you want
useEffect(() => {
  console.log("Rendered!");
});

// Runs ONCE on mount (empty dependency array) — like componentDidMount
useEffect(() => {
  fetchData();
}, []);

// Runs when 'count' changes
useEffect(() => {
  document.title = `Count: ${count}`;
}, [count]);

// Runs when userId changes — fetch new user data
useEffect(() => {
  const loadUser = async () => {
    const data = await fetchUser(userId);
    setUser(data);
  };
  loadUser();
}, [userId]);

Cleanup Function

Return a function from useEffect to clean up — clear timers, cancel subscriptions, cancel fetch requests. Runs before the component unmounts or before the effect runs again.

// Cleanup example — interval timer
useEffect(() => {
  const timer = setInterval(() => {
    setSeconds(s => s + 1);
  }, 1000);

  return () => clearInterval(timer); // cleanup on unmount
}, []);

// Cleanup example — event listener
useEffect(() => {
  const handleResize = () => setWidth(window.innerWidth);
  window.addEventListener("resize", handleResize);

  return () => window.removeEventListener("resize", handleResize);
}, []);

❌ Common Mistakes

  • Missing dependency array — runs on every render, causes infinite loops if it sets state
  • Missing dependencies — ESLint will warn you. If you use a variable inside useEffect, add it to the array
  • Setting state in useEffect without a proper dependency condition — infinite loop
  • Calling an async function directly in useEffect — define it inside and call it: const fn = async () => {}; fn();
  • Forgetting cleanup — memory leaks with timers and event listeners

Quick Summary

  • useEffect runs code after render
  • [] = run once on mount
  • [dep] = run when dep changes
  • No array = run after every render
  • Return a cleanup function
  • Can't be async — define async inside

Events in React

Handling user interactions the React way

React Events vs DOM Events

React uses synthetic events — they work exactly like DOM events but React wraps them so they work consistently across every browser. Two key differences from vanilla JS: event names are camelCase (onClick not onclick), and you always pass a function reference, never a string or a function call.

Real World Example

Think of a "Like" button on a social media post. When you click it, the like count increments and the icon turns red. That click triggers an event → the event handler runs → state updates → React re-renders the button with the new count and color. That's the complete event flow.

// ── All common event types ──────────────────────────────
<button onClick={handleClick}>Click Me</button>
<input  onChange={handleChange} />        // fires on every keystroke
<form   onSubmit={handleSubmit}>...</form>
<input  onFocus={handleFocus} />          // when input is focused
<input  onBlur={handleBlur} />            // when input loses focus
<input  onKeyDown={handleKey} />          // key pressed
<div    onMouseEnter={handleHover} />     // mouse enters
<div    onMouseLeave={handleLeave} />     // mouse leaves
<img    onLoad={handleLoaded} />          // image finished loading

// ── Passing a separate handler function ─────────────────
const handleClick = () => {
  console.log("Clicked!");
};
<button onClick={handleClick}>Click</button>   // ✅ correct
<button onClick={handleClick()}>Click</button> // ❌ calls on render!

// ── Inline arrow for passing arguments ──────────────────
<button onClick={() => deleteItem(item.id)}>Delete</button>
<button onClick={() => setCount(count + 1)}>+</button>
<button onClick={() => setTheme("dark")}>Dark Mode</button>

The Event Object

React passes the event object e into your handler automatically. Use it to read the clicked element, stop default browser behavior, or get the value of an input field.

// Reading input value
const [name, setName] = useState("");
const handleChange = (e) => {
  setName(e.target.value); // e.target is the input element
};
<input value={name} onChange={handleChange} placeholder="Your name" />

// Stopping default behavior (essential for forms)
const handleSubmit = (e) => {
  e.preventDefault(); // stops page reload
  console.log("Submitted:", name);
};
<form onSubmit={handleSubmit}>...</form>

// Reading keyboard key
const handleKey = (e) => {
  if (e.key === "Enter") submitForm();
  if (e.key === "Escape") closeModal();
};
<input onKeyDown={handleKey} />

// Getting data from the clicked element
const handleItemClick = (e) => {
  const id = e.target.dataset.id; // reads data-id attribute
  selectItem(id);
};
<li data-id={item.id} onClick={handleItemClick}>{item.name}</li>

Building a Real Interactive Component

Here's how events, state, and UI all connect together in a real component:

import { useState } from 'react';

const LikeButton = ({ initialLikes = 0 }) => {
  const [likes,   setLikes]   = useState(initialLikes);
  const [liked,   setLiked]   = useState(false);
  const [message, setMessage] = useState("");

  const handleLike = () => {
    if (liked) {
      setLikes(likes - 1);
      setLiked(false);
      setMessage("");
    } else {
      setLikes(likes + 1);
      setLiked(true);
      setMessage("Thanks for the like! ❤️");
    }
  };

  return (
    <div>
      <button
        onClick={handleLike}
        className={liked ? "btn-liked" : "btn-like"}
      >
        {liked ? "❤️ Liked" : "🤍 Like"} ({likes})
      </button>
      {message && <p>{message}</p>}
    </div>
  );
};

❌ Common Mistakes

  • Calling the function: onClick={handleClick()} — the () means "run now". It fires immediately on render, not on click. Remove the parentheses: onClick={handleClick}
  • Using lowercase: onclick — React ignores it silently. Must be camelCase onClick
  • Forgetting e.preventDefault() on form submit — the page reloads and you lose all state
  • Putting complex logic directly in JSX: onClick={() => { /* 10 lines */ }} — extract to a named handler function for readability
  • Recreating handler functions in a loop without useCallback — can hurt performance

Quick Summary

  • Events are camelCase: onClick, onChange, onSubmit
  • Pass function reference: onClick={fn} not onClick={fn()}
  • Use arrow function to pass arguments: onClick={() => fn(id)}
  • e.target.value reads input field content
  • Always e.preventDefault() on form submit
  • onChange fires on every single keystroke

Lists & Keys

Rendering arrays of data — used in every single React app

Rendering Lists with map()

In React, whenever you have an array of data you want to display (products, users, messages, todos), you use .map() to transform each item into a JSX element. React then renders all those elements to the page. This is the most common pattern in all of React.

Real World Example

Think of Spotify's playlist view — there are hundreds of songs, each displayed as a row with a number, title, artist, album, and duration. That data comes from an array. React maps over every song object and renders the same SongRow component for each one, just with different props.

// Basic list rendering
const products = [
  { id: 1, name: "Shoes",  price: 59.99 },
  { id: 2, name: "Shirt",  price: 29.99 },
  { id: 3, name: "Hat",    price: 19.99 },
];

const ProductList = () => (
  <ul>
    {products.map(product => (
      <li key={product.id}>
        {product.name} — ${product.price}
      </li>
    ))}
  </ul>
);

// Better — render a component per item (most common in real apps)
const ProductList = () => (
  <div className="grid">
    {products.map(product => (
      <ProductCard
        key={product.id}
        name={product.name}
        price={product.price}
      />
    ))}
  </div>
);

// Even cleaner with spread — passes all object properties as props
const ProductList = () => (
  <div className="grid">
    {products.map(product => (
      <ProductCard key={product.id} {...product} />
    ))}
  </div>
);

Why keys matter — the detailed explanation

When your list changes (item added, removed, reordered), React needs to figure out what actually changed so it only updates those parts of the DOM. Without keys, React has to tear down and rebuild the entire list. With stable unique keys, React can say "this specific item moved to position 3, update only its position" — much faster and avoids losing input focus or animation state.

// ❌ WRONG — index as key breaks on reorder/delete
{todos.map((todo, index) => (
  <TodoItem key={index} todo={todo} />  // DON'T do this
))}

// ✅ CORRECT — stable unique ID from data
{todos.map(todo => (
  <TodoItem key={todo.id} todo={todo} />
))}

// ─────────────────────────────────────────────
// Combining filter + map (very common pattern)
const [filter, setFilter] = useState("all");

const visibleTodos = todos
  .filter(todo => filter === "all" || todo.status === filter)
  .map(todo => <TodoItem key={todo.id} todo={todo} />);

return (
  <div>
    <div className="filters">
      <button onClick={() => setFilter("all")}>All</button>
      <button onClick={() => setFilter("done")}>Done</button>
      <button onClick={() => setFilter("active")}>Active</button>
    </div>
    <ul>{visibleTodos}</ul>
    {visibleTodos.length === 0 && <p>No items to show</p>}
  </div>
);

❌ Common Mistakes

  • Using array index as key — when you delete item 2, item 3 becomes the new index 2. React thinks you updated item 2, not deleted it. Causes wrong re-renders and lost input state
  • Forgetting the key prop entirely — React shows a warning in the console and re-renders the whole list on every change
  • Putting key on the wrong element — it must go on the outermost JSX element returned inside the map, not on a child inside it
  • Using non-unique keys — if two items share the same key, React gets confused and renders incorrectly

Quick Summary

  • Use .map() to turn arrays into JSX
  • Every item needs a key prop
  • Key must be unique and stable — use data IDs
  • Never use array index as key
  • Chain .filter().map() for filtered lists
  • Show empty state when list length is 0

Conditional Rendering

Showing and hiding UI based on state — used in every component

What is Conditional Rendering?

In React, you often need to show different UI depending on state — show a spinner while loading, show an error if something failed, show a welcome message if logged in, hide an admin panel if not an admin. This is conditional rendering — deciding what to show based on a condition. There are 4 main patterns and each one is used for a different situation.

Real World Example

Think of a YouTube video page. While the video loads → show a skeleton loader. If the video fails → show "Video unavailable". If you're not logged in → show "Sign in to comment". If you're the video owner → show an "Edit" button. All of that is conditional rendering based on different state values.

Pattern 1 — Early Return (for loading / error states)

The cleanest pattern. Check for edge cases at the top of the component and return early. This leaves the main return clean and focused on the happy path.

const UserProfile = ({ userId }) => {
  const [user,    setUser]    = useState(null);
  const [loading, setLoading] = useState(true);
  const [error,   setError]   = useState(null);

  useEffect(() => {
    fetchUser(userId).then(setUser).catch(setError).finally(() => setLoading(false));
  }, [userId]);

  // Handle each state with an early return
  if (loading) return <div className="spinner">Loading...</div>;
  if (error)   return <div className="error">Failed to load user: {error.message}</div>;
  if (!user)   return <div>User not found</div>;

  // Happy path — all good, render the real UI
  return (
    <div className="profile">
      <img src={user.avatar} alt={user.name} />
      <h2>{user.name}</h2>
      <p>{user.bio}</p>
    </div>
  );
};

Pattern 2 — Ternary (show one of two things)

Use when you have exactly two options — show A or show B. Perfect for toggle buttons, login/logout states, or switching between two views.

// Toggle text
<button onClick={toggleLogin}>
  {isLoggedIn ? "Log Out" : "Log In"}
</button>

// Toggle whole UI section
{isLoggedIn
  ? <UserDashboard user={user} />
  : <LoginPrompt />
}

// Toggle icon
<span>{isFavorited ? "❤️" : "🤍"}</span>

Pattern 3 — && Operator (show or show nothing)

Use when you want to show something only if a condition is true, and show nothing at all if it's false. It's shorter than a ternary with null on the else side.

// Show admin panel only for admins
{isAdmin && <AdminPanel />}

// Show error message only if there's an error
{error && <p className="error-msg">{error}</p>}

// Show empty state only when list is empty
{todos.length === 0 && <p className="empty">No todos yet! Add one above.</p>}

// Show notification badge only when count > 0
{unreadCount > 0 && <span className="badge">{unreadCount}</span>}

// ⚠️ Watch out — 0 is falsy but renders as "0" not nothing
// BAD:
{items.length && <List />}   // renders "0" when empty!
// GOOD:
{items.length > 0 && <List />} // renders nothing when empty

Pattern 4 — Conditional className

Change the appearance of an element by conditionally adding CSS classes based on state — without hiding or showing the element.

// Simple toggle class
<button className={isActive ? "btn btn-active" : "btn"}>
  Click Me
</button>

// Template literal (cleaner)
<button className={`btn ${isActive ? "active" : ""}`}>
  Click Me
</button>

// Multiple conditional classes
<div className={`card ${isDark ? "dark" : "light"} ${isSelected ? "selected" : ""}`}>
  Content
</div>

// Using an array + filter + join (for many classes)
<button
  className={[
    "btn",
    isActive  ? "active"   : "",
    isLoading ? "loading"  : "",
    isDisabled? "disabled" : "",
  ].filter(Boolean).join(" ")}
>
  Submit
</button>

❌ Common Mistakes

  • Using if/else directly inside JSX return — JSX only accepts expressions, not statements. Use ternary or early return instead
  • Using {count && <Thing />} when count could be 0 — renders the number "0" on screen. Always use {count > 0 && ...}
  • Deeply nested ternaries — a ? b : c ? d : e is unreadable. Use early returns or a variable instead
  • Forgetting to handle the loading state — UI flickers or crashes when data hasn't arrived yet

Quick Summary

  • Early return — for loading, error, empty states
  • Ternary ? : — show A or show B
  • && — show something or show nothing
  • Conditional className — change appearance
  • Never use if/else inside JSX return
  • Watch out for 0 && — use > 0 && instead

Forms & Controlled Inputs

The React way to handle user input

Controlled vs Uncontrolled

In React, form inputs should be controlled — their value comes from state, and every change updates state. React is the single source of truth. This lets you validate, transform, and access input values at any time.

const SignupForm = () => {
  const [formData, setFormData] = useState({
    name: "", email: "", password: ""
  });
  const [errors, setErrors] = useState({});

  // Generic handler for all fields
  const handleChange = (e) => {
    const { name, value } = e.target;
    setFormData(prev => ({ ...prev, [name]: value }));
  };

  const validate = () => {
    const errs = {};
    if (!formData.name.trim())          errs.name = "Name is required";
    if (!formData.email.includes("@"))  errs.email = "Invalid email";
    if (formData.password.length < 6)   errs.password = "Min 6 characters";
    return errs;
  };

  const handleSubmit = (e) => {
    e.preventDefault();
    const errs = validate();
    if (Object.keys(errs).length > 0) {
      setErrors(errs);
      return;
    }
    console.log("Submit:", formData);
  };

  return (
    <form onSubmit={handleSubmit}>
      <input name="name"     value={formData.name}     onChange={handleChange} placeholder="Name" />
      {errors.name && <p className="error">{errors.name}</p>}

      <input name="email"    value={formData.email}    onChange={handleChange} placeholder="Email" />
      {errors.email && <p className="error">{errors.email}</p>}

      <input name="password" value={formData.password} onChange={handleChange} type="password" placeholder="Password" />
      {errors.password && <p className="error">{errors.password}</p>}

      <button type="submit">Sign Up</button>
    </form>
  );
};

❌ Common Mistakes

  • Forgetting value={state} — uncontrolled input, React loses track of it
  • Forgetting name attribute when using a generic handler
  • Forgetting e.preventDefault() — page reloads on submit
  • Validating only on submit — validate on blur for better UX

  • Validate: required, email format, password min length, passwords match

  • Show a success message on valid submit
Goal: Controlled forms + validation is in every real web app. Master this pattern.

Lifting State Up

Sharing state between sibling components

The Problem — Siblings Can't Talk to Each Other

When two sibling components need to share the same data, you can't put the state in either one — neither component can read the other's state directly. State only flows down via props, never sideways.

The solution is simple: move the state up to the closest common parent. The parent owns the state and passes it down to both children as props. One child reads it, the other changes it — but both get it from the same source.

This is called lifting state up, and it's the #1 core React pattern for sharing data between components.

Real World Example — E-commerce Search

Think of a shopping page: there's a SearchBar at the top and a ProductList below. When you type in the search bar, the product list must filter. These two components are siblings — neither can talk to the other directly. The parent ShopPage must own the search query.

// ❌ WRONG — siblings can't share state
const ShopPage = () => (
  <div>
    <SearchBar />    {/* owns its own searchQuery — no one else can see it */}
    <ProductList />  {/* needs searchQuery — but can't reach it! */}
  </div>
);

// ✅ CORRECT — lift state up to the parent
const ShopPage = () => {
  const [searchQuery, setSearchQuery] = useState("");  // ← state lives here

  return (
    <div>
      {/* pass state down + pass the setter as a callback */}
      <SearchBar query={searchQuery} onSearch={setSearchQuery} />
      <ProductList filter={searchQuery} />
    </div>
  );
};

// SearchBar: reads the value, calls the callback to change it
const SearchBar = ({ query, onSearch }) => (
  <input
    value={query}
    onChange={e => onSearch(e.target.value)}
    placeholder="Search products..."
  />
);

// ProductList: receives the query as a prop and filters its list
const ProductList = ({ filter }) => {
  const products = [
    { id: 1, name: "React T-Shirt" },
    { id: 2, name: "JavaScript Hoodie" },
    { id: 3, name: "CSS Coffee Mug" },
  ];

  const filtered = products.filter(p =>
    p.name.toLowerCase().includes(filter.toLowerCase())
  );

  return (
    <ul>
      {filtered.length === 0
        ? <p>No products found.</p>
        : filtered.map(p => <li key={p.id}>{p.name}</li>)
      }
    </ul>
  );
};

The Golden Rule — Data Down, Events Up

This pattern follows one simple rule:

  • Data flows down — parent passes state as props to children
  • Events flow up — children call a callback prop to tell the parent to change the state

The parent is always in control of the state. Children only read it or request a change — they never own it.

// Another common example — a counter shared between two siblings
const App = () => {
  const [count, setCount] = useState(0);  // shared state lives here

  return (
    <div>
      <CounterDisplay count={count} />          {/* reads */}
      <CounterButtons onIncrement={() => setCount(c => c + 1)}
                      onDecrement={() => setCount(c => c - 1)}
                      onReset={() => setCount(0)} /> {/* changes */}
    </div>
  );
};

const CounterDisplay = ({ count }) => (
  <h2>Count: {count}</h2>
);

const CounterButtons = ({ onIncrement, onDecrement, onReset }) => (
  <div>
    <button onClick={onDecrement}>−</button>
    <button onClick={onReset}>Reset</button>
    <button onClick={onIncrement}>+</button>
  </div>
);

❌ Common Mistakes

  • Trying to pass state from a child to a sibling directly — you can't. Always lift up to the parent first
  • Lifting state too high — if only two components need it, don't put it in the root App. Find the closest common ancestor
  • Passing the entire setter function when you should wrap it — expose a named callback like onSearch instead of setQuery. Makes the API cleaner and easier to change later
  • Prop drilling — lifting state several levels deep means passing props through many layers. At that point, consider React Context instead
  • Forgetting that objects/arrays need new references — setState(arr) after mutating arr won't re-render. Always create a new array/object

Quick Summary

  • Siblings can't share state — move it to the closest common parent
  • Parent owns the state, children receive it as props
  • Pass the setter (or a wrapper callback) down so children can update it
  • Rule: data flows down, events flow up
  • Too many levels of lifting → use React Context
  • This is the most important React architecture pattern

useRef

Direct DOM access and storing values without re-rendering

What is useRef?

useRef gives you a special box — an object with a single property called .current — that you can read and write to at any time. The key thing that makes it different from state: changing .current does NOT cause a re-render.

This makes useRef perfect for two things:

  • Accessing a real DOM element directly — like focusing an input, scrolling to a section, or reading an element's size
  • Storing a value between renders — like a timer ID, an interval, or the previous value of a state — without triggering unnecessary re-renders

Real World Example — YouTube-style Video Player

A video player needs to control the real <video> DOM element — play, pause, seek. You can't do this with state alone. You need a direct reference to the element. That's exactly what useRef is for.

import { useRef, useState } from 'react';

const VideoPlayer = () => {
  const videoRef = useRef(null);   // will point to the <video> element
  const [isPlaying, setIsPlaying] = useState(false);

  const togglePlay = () => {
    if (isPlaying) {
      videoRef.current.pause();    // call the real DOM method
    } else {
      videoRef.current.play();     // call the real DOM method
    }
    setIsPlaying(!isPlaying);
  };

  return (
    <div>
      <video ref={videoRef} src="/video/intro.mp4" width="400" />
      <button onClick={togglePlay}>
        {isPlaying ? "⏸ Pause" : "▶ Play"}
      </button>
    </div>
  );
};

Use Case 1 — Auto-focus an Input

When a modal opens or a search page loads, you often want the text field to be automatically focused so the user can type right away — just like how Google search focuses the input when you open the page.

const SearchModal = ({ isOpen }) => {
  const inputRef = useRef(null);

  useEffect(() => {
    if (isOpen) {
      inputRef.current.focus();  // auto-focus when modal opens
    }
  }, [isOpen]);  // runs every time isOpen changes

  return (
    <div className={`modal ${isOpen ? "open" : ""}`}>
      <input ref={inputRef} placeholder="Search..." />
    </div>
  );
};

Use Case 2 — Store a Timer ID

If you store a setInterval timer ID in state, every time it changes it re-renders — which is wasteful and can cause bugs. Store it in a ref instead. The component re-renders only when the visible count changes, not when you save the timer ID.

const Stopwatch = () => {
  const [seconds, setSeconds] = useState(0);
  const timerRef = useRef(null);  // stores the interval ID — not in state!

  const start = () => {
    if (timerRef.current) return;  // prevent double-start
    timerRef.current = setInterval(() => {
      setSeconds(s => s + 1);
    }, 1000);
  };

  const stop = () => {
    clearInterval(timerRef.current);
    timerRef.current = null;
  };

  const reset = () => {
    stop();
    setSeconds(0);
  };

  return (
    <div>
      <p className="timer">{seconds}s</p>
      <button onClick={start}>▶ Start</button>
      <button onClick={stop}>⏹ Stop</button>
      <button onClick={reset}>↺ Reset</button>
    </div>
  );
};

Use Case 3 — Track the Previous Value

Sometimes you need to compare a value to what it was on the last render — like showing a price change indicator (went up or down). You can't do this with state alone, but a ref remembers the last render's value.

const PriceDisplay = ({ price }) => {
  const prevPriceRef = useRef(price);  // starts as current price

  useEffect(() => {
    prevPriceRef.current = price;  // update AFTER every render
  });  // no deps array — runs after every render

  const prevPrice = prevPriceRef.current;
  const direction = price > prevPrice ? "↑" : price < prevPrice ? "↓" : "→";
  const color     = price > prevPrice ? "green" : price < prevPrice ? "red" : "gray";

  return (
    <div>
      <span style={{ color }}>{direction}</span>
      <strong> ${price.toFixed(2)}</strong>
      <small> (was ${prevPrice.toFixed(2)})</small>
    </div>
  );
};

❌ Common Mistakes

  • Using a ref instead of state for values that should show in the UI — if you change ref.current, React does NOT re-render, so the screen won't update. If it needs to be visible, use useState
  • Accessing ref.current before the component mounts — on the first render the ref is null. Always check if (ref.current) or use it inside useEffect
  • Using a ref to store things that should trigger re-renders — refs are for side-effect values (timer IDs, DOM nodes). Don't use them to hide state from React
  • Forgetting to clear intervals/timeouts stored in refs — always call clearInterval / clearTimeout in the useEffect cleanup or on component unmount
  • Putting a ref in the dependency array of useEffect — refs are stable (same object every render), so listing one as a dep does nothing. The array checks the ref object, not .current

Quick Summary

  • useRef() returns a mutable { current: value } object
  • Attach to a DOM element with the ref prop — access it via myRef.current
  • Changing .current does NOT cause a re-render
  • Use Case 1 — DOM access: focus, scroll, play/pause, read dimensions
  • Use Case 2 — Store values: timer IDs, intervals, socket connections
  • Use Case 3 — Previous values: compare current vs last render
  • If the value needs to show in the UI → use useState instead

Custom Hooks

Extract reusable logic out of components

What is a Custom Hook?

A custom hook is a function that starts with "use" and contains hook calls. It lets you extract stateful logic from a component so it can be reused across multiple components — without duplicating code.

// Without custom hook — duplicated in every component
const ProductPage = () => {
  const [data, setData]       = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError]     = useState(null);

  useEffect(() => {
    fetch("/api/products").then(r => r.json())
      .then(setData).catch(setError).finally(() => setLoading(false));
  }, []);
  // ...
};

// ─── Extract to a custom hook ───────────────────────────
// hooks/useFetch.js
const useFetch = (url) => {
  const [data, setData]       = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError]     = useState(null);

  useEffect(() => {
    setLoading(true);
    fetch(url).then(r => r.json())
      .then(setData).catch(setError).finally(() => setLoading(false));
  }, [url]);

  return { data, loading, error };
};

// Now reuse it anywhere — clean!
const ProductPage = () => {
  const { data, loading, error } = useFetch("/api/products");
  if (loading) return <Spinner />;
  if (error)   return <Error />;
  return <ProductList items={data} />;
};

const UserPage = () => {
  const { data: user, loading } = useFetch("/api/user/1");
  // ...
};

Other Useful Custom Hooks

These three patterns appear in almost every real React codebase. Build them once and reuse them everywhere.

// useLocalStorage — state that survives page refresh
const useLocalStorage = (key, initial) => {
  const [value, setValue] = useState(
    () => JSON.parse(localStorage.getItem(key)) ?? initial
  );
  const setStored = (val) => {
    setValue(val);
    localStorage.setItem(key, JSON.stringify(val));
  };
  return [value, setStored];
};

// Usage — just like useState, but persisted
const [theme, setTheme] = useLocalStorage("theme", "dark");

// ─────────────────────────────────────────────────────────
// useDebounce — wait until user stops typing before acting
const useDebounce = (value, delay) => {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const timer = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(timer);  // clear on next keystroke
  }, [value, delay]);
  return debounced;
};

// Usage — only search after 400ms pause in typing
const [search, setSearch] = useState("");
const debouncedSearch = useDebounce(search, 400);
useEffect(() => { fetchResults(debouncedSearch); }, [debouncedSearch]);

// ─────────────────────────────────────────────────────────
// useWindowSize — track viewport dimensions reactively
const useWindowSize = () => {
  const [size, setSize] = useState({
    width: window.innerWidth,
    height: window.innerHeight,
  });
  useEffect(() => {
    const handler = () => setSize({ width: window.innerWidth, height: window.innerHeight });
    window.addEventListener("resize", handler);
    return () => window.removeEventListener("resize", handler);
  }, []);
  return size;
};

// Usage
const { width } = useWindowSize();
const isMobile = width < 768;

❌ Common Mistakes

  • Not starting the name with "use" — React won't enforce the rules of hooks inside it, and ESLint hooks plugin won't catch bugs
  • Putting hook calls inside conditions inside a custom hook — hooks must always be called in the same order, even inside custom hooks
  • Returning too much — only return what the component actually needs. A hook that returns 10 things is a sign it should be split into smaller hooks
  • Forgetting to clean up — if your custom hook sets up a listener, timer, or subscription, return a cleanup function from useEffect
  • Re-creating the hook logic inside a component instead of using the hook — if you find yourself copy-pasting the same useState + useEffect block, extract it

Quick Summary

  • Custom hooks = regular functions that start with "use"
  • Can call any built-in React hook inside
  • Extract repeated stateful logic from components
  • Store in a hooks/ folder, name file useHookName.js
  • Return exactly what the component needs (values + setters)
  • Always add cleanup in useEffect if you subscribe to anything

Fetching Data in React

The complete pattern — loading, data, error states

The Complete Fetch Pattern

Every real React app fetches data. The pattern always has 3 states: loading, data, and error. Always handle all three — a good UI shows a spinner while loading and a clear message when something goes wrong.

import { useState, useEffect } from 'react';

const UserDirectory = () => {
  const [users,   setUsers]   = useState([]);
  const [loading, setLoading] = useState(true);
  const [error,   setError]   = useState(null);
  const [search,  setSearch]  = useState("");

  useEffect(() => {
    const fetchUsers = async () => {
      try {
        setLoading(true);
        setError(null);
        const res = await fetch("https://jsonplaceholder.typicode.com/users");
        if (!res.ok) throw new Error(`Error: ${res.status}`);
        const data = await res.json();
        setUsers(data);
      } catch (err) {
        setError(err.message);
      } finally {
        setLoading(false);
      }
    };
    fetchUsers();
  }, []);

  const filtered = users.filter(u =>
    u.name.toLowerCase().includes(search.toLowerCase())
  );

  if (loading) return <div className="spinner">Loading...</div>;
  if (error)   return <div className="error">Error: {error}</div>;

  return (
    <div>
      <input
        value={search}
        onChange={e => setSearch(e.target.value)}
        placeholder="Search users..."
      />
      <div className="grid">
        {filtered.map(user => (
          <div key={user.id} className="card">
            <h3>{user.name}</h3>
            <p>{user.email}</p>
            <p>{user.address.city}</p>
          </div>
        ))}
      </div>
      {filtered.length === 0 && <p>No users found for "{search}"</p>}
    </div>
  );
};

Quick Summary

  • Always have loading + data + error state
  • Fetch inside useEffect([], [])
  • Define async function inside effect, call it
  • Check response.ok before parsing
  • Show spinner while loading
  • Extract to useFetch hook when reused
