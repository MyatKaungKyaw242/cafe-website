1

JSX — Fragment shorthand saves wrapper divs

Topic 1: JSX

Tip

Beginners add wrapper <div> everywhere just to satisfy the "one root element" rule. These extra divs break your CSS layout (especially flex/grid) and pollute the DOM. Use the Fragment shorthand <></> instead — it satisfies the rule without adding any element to the DOM.

// ❌ Extra div breaks your grid layout
return (
  <div>  {/* unwanted wrapper */}
    <Sidebar />
    <Main />
  </div>
);

// ✅ Fragment — no DOM element, no layout breakage
return (
  <>
    <Sidebar />
    <Main />
  </>
);

// Use <React.Fragment key={id}> when you need a key prop (e.g. in .map())
{items.map(item => (
  <React.Fragment key={item.id}>
    <dt>{item.term}</dt>
    <dd>{item.def}</dd>
  </React.Fragment>
))}
Rule: Reach for <></> before reaching for <div>. Add a div only when you actually need it for styling.

Components — extract when you copy-paste JSX twice

Topic 2: Components

Trick

The hardest skill in React is knowing when to extract a component. The simplest rule: if you write the same JSX block twice, extract it. Don't over-engineer early — start in one file, extract when you feel the pain of repetition.

// You write this twice:
<div className="user-card">
  <img src={user.avatar} />
  <h3>{user.name}</h3>
  <p>{user.bio}</p>
</div>

// That's the signal — extract it
const UserCard = ({ user }) => (
  <div className="user-card">
    <img src={user.avatar} />
    <h3>{user.name}</h3>
    <p>{user.bio}</p>
  </div>
);

// Now use it
<UserCard user={author} />
<UserCard user={commenter} />
Rule: One copy = fine. Two copies = extract. Three copies = you're late.

Props — optional chaining prevents crashes on missing data

Topic 3: Props

Tip

When a prop might be null or undefined — for example, data from an API that hasn't loaded yet — accessing a property on it crashes the app. Optional chaining ?. safely returns undefined instead of throwing an error. Combine with ?? to provide a fallback.

// ❌ Crashes if user is null
const Profile = ({ user }) => (
  <div>
    <h2>{user.name}</h2>        // TypeError: Cannot read property of null
    <p>{user.address.city}</p>  // crashes if address is missing
  </div>
);

// ✅ Optional chaining — safe
const Profile = ({ user }) => (
  <div>
    <h2>{user?.name ?? "Anonymous"}</h2>
    <p>{user?.address?.city ?? "Unknown city"}</p>
    <a href={user?.website}>{user?.website ?? "No website"}</a>
  </div>
);

// Also works for callback props — call only if provided
<button onClick={() => onDelete?.()}>Delete</button>
Rule: Use ?. whenever a prop could be undefined. Use ?? to show a fallback value.
4

useState — always use functional updates for derived state

Topic 4: useState

Watch Out

When your new state depends on the old state (like incrementing a counter or adding to an array), always use the functional update form setState(prev => ...). Using the value directly can cause a stale closure bug — especially inside event handlers or effects where the state value was captured at an earlier render.

// ❌ Stale closure bug — both clicks read the same old value
const handleDoubleClick = () => {
  setCount(count + 1); // count is stale
  setCount(count + 1); // still the same count!
  // Result: only increments by 1, not 2
};

// ✅ Functional update — always gets the latest value
const handleDoubleClick = () => {
  setCount(prev => prev + 1); // gets current count
  setCount(prev => prev + 1); // gets UPDATED count from previous call
  // Result: increments by 2 ✓
};

// Same for arrays
setItems(prev => [...prev, newItem]);   // ✅
setItems([...items, newItem]);          // ❌ stale if called twice fast
Rule: If the new state is based on the old state, always use setState(prev => newValue).
5

useEffect — the dependency array is not optional

Topic 5: useEffect

Watch Out

The three forms of useEffect do very different things. Forgetting the deps array causes infinite loops. An empty array [] means "run once". A filled array means "run when these values change". Missing a dependency causes stale data bugs.

// No array — runs after EVERY render (usually not what you want)
useEffect(() => { console.log("rendered"); });

// Empty array [] — runs ONCE on mount
useEffect(() => { fetchInitialData(); }, []);

// With dependency — runs when userId changes
useEffect(() => { fetchUser(userId); }, [userId]);

// ⚠️ Missing dependency — stale data bug
const [id, setId] = useState(1);
useEffect(() => {
  fetchUser(id); // id used but not listed in deps
}, []);          // ❌ will always use the initial id=1

useEffect(() => {
  fetchUser(id);
}, [id]);  // ✅ re-runs whenever id changes
Rule: Every variable from the component that's used inside a useEffect must be in the dependency array.
6

Component Lifecycle — always clean up subscriptions and timers

Topic 6: Component Lifecycle

Watch Out

When a component unmounts (is removed from the screen), any timers or event listeners you set up keep running — they don't auto-stop. This causes memory leaks and "can't update state on unmounted component" errors. Always return a cleanup function from useEffect.

// ❌ Timer keeps running after component is removed
useEffect(() => {
  const timer = setInterval(fetchNewData, 5000);
  // no cleanup — timer runs forever!
}, []);

// ✅ Return cleanup function — runs on unmount
useEffect(() => {
  const timer = setInterval(fetchNewData, 5000);
  return () => clearInterval(timer); // stops the timer
}, []);

// ✅ Event listener cleanup
useEffect(() => {
  window.addEventListener("resize", handleResize);
  return () => window.removeEventListener("resize", handleResize);
}, []);

// ✅ AbortController for fetch cleanup
useEffect(() => {
  const controller = new AbortController();
  fetch(url, { signal: controller.signal }).then(/* ... */);
  return () => controller.abort();
}, [url]);
Rule: If your useEffect sets something up, the cleanup function must tear it down.
7

Styling — CSS Modules prevent class name collisions

Topic 7: Styling in React

Trick

Global CSS class names clash when your app grows — two components both using .card will fight over the styles. CSS Modules (built into Vite) automatically generate unique class names per file so there are never any collisions. Zero extra setup needed.

// Button.module.css
.btn {
  padding: 10px 20px;
  background: #61dafb;
  border-radius: 8px;
}

.btn:hover {
  background: #48b8d8;
}

// Button.jsx — import as an object
import styles from './Button.module.css';

const Button = ({ label }) => (
  <button className={styles.btn}>
    {label}
  </button>
);
// Vite compiles .btn to "Button_btn__xKj3a" — unique!
// No global class collision, ever.
Rule: For small projects, regular CSS is fine. For anything growing, use CSS Modules — it's free and already included in Vite.
8

Events — don't call the function, pass it

Topic 8: Events

Watch Out

The most common beginner mistake with events: adding () after the function name. This calls the function immediately when the component renders — not when the user clicks. You want to pass a reference to be called later.

// ❌ WRONG — handleClick() runs immediately on render!
<button onClick={handleClick()}>Click</button>

// ✅ CORRECT — pass the function, React calls it on click
<button onClick={handleClick}>Click</button>

// When you need to pass arguments, wrap in an arrow function
<button onClick={() => handleDelete(item.id)}>Delete</button>

// ❌ WRONG — this calls handleDelete(item.id) immediately
<button onClick={handleDelete(item.id)}>Delete</button>
Rule: onClick={fn} — no parentheses. onClick={() => fn(arg)} — arrow wrapper for arguments.
9

Lists & Keys — index as key causes subtle bugs

Topic 9: Lists & Keys

Watch Out

Using the array index as a key works until you sort, filter, or delete items. React uses keys to track which items changed — if you delete item 0, every remaining item shifts index. React thinks item 1 is now item 0 and gets confused, causing wrong component state, broken animations, or duplicate data.

// ❌ Index as key — breaks on delete/reorder
{items.map((item, index) => (
  <Item key={index} {...item} />
))}

// ✅ Stable unique ID as key
{items.map(item => (
  <Item key={item.id} {...item} />
))}

// If your data has no ID, create one when you add it:
const addItem = (text) => {
  setItems(prev => [
    ...prev,
    { id: crypto.randomUUID(), text }  // stable unique id
  ]);
};

// When index is OK: static lists that never reorder, filter, or delete
Rule: Use the item's database ID as the key. If there's no ID, generate one with crypto.randomUUID() when the item is created.
10

Conditional Rendering — the 0 && gotcha will bite you

Topic 10: Conditional Rendering

Watch Out

This is one of the most common React bugs. When you write {count && <Badge />}, if count is 0, React renders the number 0 on the screen — not nothing. That's because 0 is falsy but it's still a valid JSX value. Always coerce to a boolean.

// ❌ Renders "0" on screen when count is 0
{count && <Badge count={count} />}

// ✅ Convert to boolean first
{count > 0 && <Badge count={count} />}
{!!count && <Badge count={count} />}
{Boolean(count) && <Badge count={count} />}

// Same trap with array length
{items.length && <List />}     // ❌ renders "0" when empty!
{items.length > 0 && <List />} // ✅

// Also be careful with undefined — always fine (renders nothing)
// null — fine (renders nothing)
// 0, "", false — fine in ternary, dangerous in &&
Rule: Never use {number && ...}. Always use {number > 0 && ...}.
11

Forms — validate on blur, not just on submit

Topic 11: Forms & Controlled Inputs

Tip

Validating only on submit makes for a bad UX — the user fills in everything and hits submit, then gets 5 errors at once. Professional forms validate when a field loses focus (onBlur) — after the user is done with that field. Show the error inline, immediately, right under the relevant input.

const [touched, setTouched] = useState({});
const [errors, setErrors]   = useState({});

const validate = (name, value) => {
  if (name === "email" && !value.includes("@"))
    return "Invalid email address";
  if (name === "password" && value.length < 6)
    return "Minimum 6 characters";
  return "";
};

// Mark field as touched on blur, validate immediately
const handleBlur = (e) => {
  const { name, value } = e.target;
  setTouched(prev => ({ ...prev, [name]: true }));
  setErrors(prev => ({ ...prev, [name]: validate(name, value) }));
};

// Show error only after user has touched the field
{touched.email && errors.email && (
  <p className="error">{errors.email}</p>
)}
Rule: Validate onBlur for real-time feedback. Validate again onSubmit as the final check before sending data.
12

Lifting State — lift only as high as needed

Topic 12: Lifting State Up

Trick

A very common mistake is lifting state all the way to the root App component when only two sibling components need it. The further up state lives, the more components re-render when it changes, and the more prop drilling you have to do. Lift to the closest common ancestor — no higher.

// ❌ Lifted too high — App re-renders for a modal 3 levels deep
const App = () => {
  const [isModalOpen, setIsModalOpen] = useState(false);
  return <Page modal={isModalOpen} setModal={setIsModalOpen} />;
};

// ✅ Keep it at the level that actually needs it
const ProductCard = () => {
  const [isModalOpen, setIsModalOpen] = useState(false);
  return (
    <>
      <button onClick={() => setIsModalOpen(true)}>Details</button>
      {isModalOpen && <Modal onClose={() => setIsModalOpen(false)} />}
    </>
  );
};

// When multiple siblings need it → lift to their parent
// When many levels need it → use Context
Rule: Find the lowest component where the state makes sense. That's where it should live.
13

useRef — if it doesn't show on screen, it probably belongs in a ref

Topic 13: useRef

Tip

A quick mental model to decide between useState and useRef: if the value needs to show in the UI, use state. If it's behind-the-scenes infrastructure, use a ref. Timer IDs, socket connections, interval IDs, previous values — these never appear on screen, so they belong in refs, not state.

// ✅ State — because the count shows on screen
const [seconds, setSeconds] = useState(0);

// ✅ Ref — because the timer ID never shows on screen
const timerRef = useRef(null);

// ✅ State — because isPlaying affects the button label
const [isPlaying, setIsPlaying] = useState(false);

// ✅ Ref — because videoRef is just a pointer to the DOM element
const videoRef = useRef(null);

// Common ref use cases:
// - setInterval / setTimeout IDs
// - WebSocket connections
// - DOM elements (input focus, video play, scroll position)
// - Previous state values for comparison
// - Any value that's "technical" not "visual"
Rule: Ask "does the user see this value on screen?" — Yes → useState. No → useRef.
14

Custom Hooks — name them after what they do, not how they work

Topic 14: Custom Hooks

Trick

A well-named custom hook reads like English and tells you exactly what the component is doing — without you needing to see the implementation. Name it after the feature or domain (useAuth, useCart, useSearchResults), not the technical details (useStateWithFetch, useEffectAndSet).

// ❌ Named after implementation details
const useStateAndEffect = (url) => { /* ... */ };
const useSetAndGet = (key) => { /* ... */ };

// ✅ Named after what it gives you
const useFetch       = (url) => { /* ... */ };  // gives you fetched data
const useLocalStorage = (key) => { /* ... */ }; // gives you persistent state
const useDebounce    = (val) => { /* ... */ };  // gives you a debounced value
const useAuth        = ()    => { /* ... */ };  // gives you auth state
const useCart        = ()    => { /* ... */ };  // gives you cart + actions
const useMediaQuery  = (q)   => { /* ... */ };  // gives you a boolean match

// Reading component code becomes natural:
const { user, isLoggedIn } = useAuth();
const [theme, setTheme]    = useLocalStorage("theme", "dark");
const { data, loading }    = useFetch("/api/products");
Rule: Your custom hook name should describe the answer it gives, not the React hooks it uses inside.
15

Fetching Data — always check response.ok before parsing

Topic 15: Fetching Data

Watch Out

fetch() only rejects its Promise for network errors (no internet, DNS failure). A 404 or 500 HTTP response is considered a "successful" fetch — the Promise resolves. If you skip the res.ok check, your app will try to parse an error response as data and silently show nothing or crash.

// ❌ Doesn't catch 404, 500, etc. — silently fails
const res = await fetch("/api/user/999");
const data = await res.json(); // parses the 404 error body as if it's data!

// ✅ Always check res.ok
const res = await fetch("/api/user/999");
if (!res.ok) {
  throw new Error(`Server error: ${res.status} ${res.statusText}`);
}
const data = await res.json(); // only runs if 200-299

// Full pattern with specific error messages
if (res.status === 404) throw new Error("User not found");
if (res.status === 401) throw new Error("Please log in");
if (res.status === 403) throw new Error("Access denied");
if (!res.ok)            throw new Error(`Error ${res.status}`);
Rule: Always write if (!res.ok) throw new Error(...) right after every await fetch() call.
