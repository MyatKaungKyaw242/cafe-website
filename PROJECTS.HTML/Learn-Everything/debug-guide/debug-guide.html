Debugging Guide

Your code is broken. Don't panic. Work through these steps.

🐛 The best developers aren't the ones who never get bugs — they're the ones who fix them fastest
🎨

My HTML or CSS isn't working

Layout broken, styles not applying, elements in the wrong place

1

Check for typos in class names

The #1 CSS bug — a mismatch between HTML and CSS

CSS is case-sensitive. .Card and .card are different. A single letter typo means the style never applies and no error is shown anywhere.

  • In HTML: class="nav-btn"
  • In CSS: .nav-btn { ... } — must match exactly
  • Check for navbtn vs nav-btn vs nav_btn
  • Right-click the element → Inspect → check what classes are applied
Quick check: In DevTools, click the element. Does the class show in the Styles panel? If not, the class name doesn't match.
2

Check if the CSS file is actually linked

Styles don't apply at all — every element looks plain

If your whole page looks unstyled, the CSS file probably isn't loading. Check your <link> tag path.

<!-- Check the href path is correct -->
<link rel="stylesheet" href="style.css" />      <!-- same folder -->
<link rel="stylesheet" href="css/style.css" />  <!-- in a subfolder -->
<link rel="stylesheet" href="../style.css" />   <!-- one folder up -->
  • Open DevTools → Network tab → reload the page
  • Look for your CSS file — if it shows red or 404, the path is wrong
  • Check spelling — styles.css vs style.css
3

Check for specificity conflicts

Your style is being overridden by another rule

You wrote the CSS but it still doesn't show. Another rule is winning. Open DevTools → click the element → Styles panel shows all rules. Crossed-out styles are being overridden.

  • A rule with an ID (#nav) beats one with a class (.nav)
  • A rule with more selectors beats a simpler one
  • A rule that comes later in the file beats an earlier one
  • Never use !important to "force" a style — fix the specificity instead
DevTools trick: In the Styles panel, hover over a crossed-out property to see which rule is winning. Click the winning rule to jump to it.
4

Layout not behaving — check the Box Model

Margins, padding, sizes look wrong

Open DevTools → click an element → look at the Box Model diagram in the Computed tab. It shows you exactly how much space is content, padding, border, and margin.

  • Element wider than expected? Check padding — it adds to width unless you use box-sizing: border-box
  • Margin not working? Margins collapse between siblings and between parent/child vertically
  • Element overflowing its parent? Try overflow: hidden on the parent to diagnose
  • Always have * { box-sizing: border-box; } at the top of your CSS
5

Flexbox or Grid not working

Items not aligning or sizing correctly

The most common mistake: applying flex properties to the wrong element. justify-content and align-items go on the parent (container), not the children.

/* ❌ Wrong — centering on the child does nothing */
.child { justify-content: center; }

/* ✅ Correct — centering goes on the parent */
.parent {
  display: flex;
  justify-content: center;  /* horizontal */
  align-items: center;      /* vertical */
}
  • Is display: flex or display: grid on the parent? Check in DevTools
  • Grid not creating columns? Make sure grid-template-columns is set
  • Items not wrapping? Add flex-wrap: wrap

My JavaScript isn't working

Button does nothing, function not running, wrong output

1

Open the Console first — always

Most JS errors show up here as red text

Press F12 → Console tab. Every error, warning, and console.log output appears here. The error message tells you exactly what went wrong and on which line. Read it before anything else.

  • Uncaught ReferenceError: x is not defined — you used a variable that doesn't exist in that scope
  • TypeError: Cannot read properties of null — you're accessing a property on something that is null
  • SyntaxError: Unexpected token — a typo in your code structure (missing bracket, comma, etc.)
  • Click the filename:line number on the right of the error to jump to that exact line
Shortcut: Right-click anywhere on a page → Inspect → Console. Or F12 → Console. Keep it open while coding.
2

Use console.log to see what's actually happening

Your code runs but gives the wrong result

console.log is your best friend. Drop it at different points to see what value a variable has at that moment. This is called "print debugging" and it's what every developer does.

const calculateTotal = (items) => {
  console.log("items received:", items); // what did we get?

  const total = items.reduce((sum, item) => {
    console.log("current sum:", sum, "item price:", item.price);
    return sum + item.price;
  }, 0);

  console.log("final total:", total); // what did we produce?
  return total;
};
  • Log at the start of a function to confirm it's being called
  • Log the input to check you're receiving what you expect
  • Log at each step to find where the value goes wrong
  • Use console.log({ variableName }) — the curly braces show the name and value together
3

My function is never running

A click or event doesn't trigger anything

Add a console.log("function called!") as the very first line of your function. If it doesn't appear in the console when you click, the event listener is the problem.

  • Check if the element actually exists: console.log(document.getElementById("btn")) — if null, the ID is wrong or the script runs before the DOM loads
  • Is the script tag at the bottom of the body (before </body>)? If it's in <head>, the elements don't exist yet when the script runs
  • Did you accidentally call the function instead of passing it? onClick={handleClick()} calls immediately — remove the ()
  • Check for typos in the event name: "click" not "Click" or "onclick"
4

TypeError: Cannot read properties of null / undefined

The most common JS crash

You're trying to access a property or method on something that doesn't exist. The error tells you what is null/undefined and what property you tried to access.

// Error: Cannot read properties of null (reading 'innerHTML')
// Means: document.getElementById(...) returned null
document.getElementById("header").innerHTML = "Hello";
// Fix: check the ID — "header" doesn't exist in your HTML

// Error: Cannot read properties of undefined (reading 'name')
// Means: user is undefined
console.log(user.name);
// Fix: check where user comes from — it may not be set yet
console.log(user); // log it first to see what it actually is
  • Log the variable before accessing its property to see what it actually is
  • Use optional chaining: user?.name won't crash if user is undefined
  • Check if you're accessing data before it's loaded (async timing issue)
5

Check your JS file is linked correctly

Script does absolutely nothing

Same as CSS — if the file path is wrong, the script silently never loads. Check the Network tab for a red 404 error on your JS file.

<!-- Place script at the BOTTOM, before </body> -->
<script src="script.js"></script>      <!-- same folder -->
<script src="js/script.js"></script>   <!-- in a subfolder -->

<!-- If in <head>, you MUST use defer -->
<script src="script.js" defer></script>
Common mistake: Script in <head> without defer — the script runs before the HTML is parsed, so document.getElementById() returns null for everything.

My React isn't working

Component not updating, state not changing, errors in console

1

How to read a React error message

Red screen with a wall of text — here's how to decode it

React error messages look scary but they follow a pattern. Read from the top:

  • Line 1: The error type and message — this is the most important part. Read it carefully.
  • Line 2–5: The component stack — where in your component tree the error happened
  • Rest: The full call stack — usually React internals, mostly ignore these
// Example error:
TypeError: Cannot read properties of undefined (reading 'map')
  at ProductList (ProductList.jsx:12)

// What it means:
// You called .map() on something that is undefined
// Line 12 of ProductList.jsx
// Fix: the data you're trying to map over hasn't loaded yet
// Add a loading check: if (!products) return <Spinner />;
Quick tip: The error always shows you the file and line number. Click it to jump straight to the problem.
2

State changed but screen didn't update

You called setState but nothing re-rendered

React only re-renders when it sees a new value. If you mutate an array or object directly, the reference stays the same and React thinks nothing changed.

// ❌ WRONG — mutating the array directly
const addItem = (item) => {
  items.push(item);       // mutates in-place
  setItems(items);        // same reference — React doesn't re-render!
};

// ✅ CORRECT — create a new array
const addItem = (item) => {
  setItems(prev => [...prev, item]); // new array = new reference = re-render ✓
};

// ❌ WRONG — mutating an object
user.name = "New Name";
setUser(user); // same reference!

// ✅ CORRECT
setUser(prev => ({ ...prev, name: "New Name" }));
Rule: Never use push, pop, splice, or direct property assignment on state. Always create a new array or object.
3

useEffect running in an infinite loop

Page keeps re-rendering, freezes, or network tab shows endless requests

An infinite loop in useEffect means the effect runs, changes state, which triggers a re-render, which runs the effect again — forever. Two common causes:

// CAUSE 1 — No deps array (runs after every render)
useEffect(() => {
  setData(fetchedData); // this sets state
});                     // no array = runs again = infinite loop!

// FIX — Add the deps array
useEffect(() => {
  setData(fetchedData);
}, []); // ← empty array = runs once only

// CAUSE 2 — Object/array in deps array
useEffect(() => {
  doSomething(config);
}, [config]); // if config is { key: "val" }, it's a NEW object every render!

// FIX — Use a primitive value or useMemo
useEffect(() => {
  doSomething(config);
}, [config.key]); // use the primitive value inside the object
4

Component shows old/stale data

State updates don't seem to take effect immediately

React state updates are asynchronous. Calling setState doesn't change the value immediately — the new value is only available on the next render. Reading state right after setting it still gives you the old value.

// ❌ Wrong expectation — count is still 0 here
const handleClick = () => {
  setCount(count + 1);
  console.log(count); // still 0! update hasn't happened yet
};

// ✅ Use useEffect to react to state changes
useEffect(() => {
  console.log("count changed:", count); // runs after re-render with new value
}, [count]);

// ✅ Or use the functional update form if you need the latest value
setCount(prev => {
  const newCount = prev + 1;
  console.log("new count:", newCount); // correct!
  return newCount;
});
5

Common React error messages decoded

The errors you'll see most often

  • Each child in a list should have a unique "key" prop → You're using .map() without adding a key prop to each element
  • Hooks can only be called inside a function component → You called a hook outside a component or inside a loop/if statement
  • Cannot update a component while rendering a different component → You're calling setState directly in the render (not in an event handler or useEffect)
  • Objects are not valid as a React child → You tried to render a plain JS object. You must access a property: {user.name} not {user}
  • Too many re-renders → You're calling setState directly in the component body (not in an event or effect). React hits a loop.

DevTools Quick Reference

The tools that make debugging 10x faster

1

Elements Tab — inspect HTML & CSS live

See and edit your HTML/CSS in real time

  • Right-click any element → Inspect — jumps to that element in the HTML
  • Styles panel on the right — all CSS rules applied to the selected element
  • Crossed-out rules — being overridden by a more specific rule
  • Click a CSS value to edit it live — great for experimenting without touching your files
  • Computed tab — shows the final resolved values after all rules are applied
  • Box Model diagram at the bottom of the Computed tab
Shortcut: F12 → Elements, or right-click → Inspect. Changes here are temporary — they reset on refresh.
2

Console Tab — run JS and see errors

Your main debugging window

  • All console.log output appears here
  • Red text = errors (your code is broken)
  • Yellow text = warnings (things to be aware of)
  • Click the file:line link on the right of any error to jump to that code
  • You can type JavaScript directly in the console and run it — great for testing snippets
  • Type $0 to reference the currently selected element in the Elements tab
3

Network Tab — debug API calls

See every request your page makes

  • Reload the page with the Network tab open to see all requests
  • Filter by "Fetch/XHR" to see only API calls
  • Red items = failed requests (404 not found, 500 server error)
  • Click any request → Response tab to see the actual data returned
  • Click any request → Headers tab to see the URL, method, and status code
  • Status 200 = success. 404 = not found. 401 = unauthorized. 500 = server error
Most useful for: Checking if your fetch() is hitting the right URL and what data the server is actually sending back.
4

The Debugging Process — a repeatable system

How to approach any bug methodically

When your code breaks, don't randomly change things. Follow this process:

Open the Console. Read the error message carefully — what does it say, and on which line?
If no error, add console.log at the start of the function — confirm it's being called.
Log the inputs — is the function receiving what you expect?
Log at each step — where does the value first go wrong?
Isolate the problem — comment out half the code. Does the bug still happen? Keep narrowing it down.
Google the exact error message in quotes — someone has had this exact error before.
Read your code out loud — explain what each line does. You'll often spot the bug yourself.
Take a break — fresh eyes catch bugs that tired eyes miss.
Remember: Every bug has a logical cause. It's not magic. Work through it step by step and you will find it.
